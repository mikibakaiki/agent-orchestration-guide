Field note · evaluation

Evaluating obra/superpowers

A skills-and-subagents framework by Jesse Vincent that wraps your coding agent in a mandatory brainstorm → plan → TDD → subagent-execute → review workflow. It is the most complete public expression of the patterns in this guide — and also the clearest case study in their costs. Here's whether it's worth adopting, and when.

what it is, in one paragraph

Superpowers is a plugin (Claude Code, Codex, Gemini, OpenCode, Cursor, Copilot CLI) that installs a library of skills — markdown files describing procedures — plus a session-start hook that forces the agent to consult them. The skills encode a full methodology: Socratic brainstorming, worktree isolation, bite-sized planning, subagent-driven execution with two-stage review, and strict RED-GREEN-REFACTOR TDD. The agent is instructed that skills are mandatory, not suggestions.

How the "loop" actually works

This isn't a Ralph-style while loop. It's a skill-gated state machine the agent walks through, with a subagent dispatched per task. Mapped onto the four patterns from chapter 04:

brainstorming — prompt chaining

Before any code, the agent refuses to implement and instead interrogates you to extract a spec, presenting it in chunks small enough to actually read. Output: a saved design document. This is plan-mode with teeth.

using-git-worktrees — isolation

On design approval, it auto-creates a worktree on a new branch, runs setup, verifies a clean test baseline. Exactly the pattern from step 5, automated.

writing-plans — compression

Breaks work into 2–5 minute tasks, each with exact file paths and complete code, written for "an enthusiastic junior engineer with poor taste, no judgement, and an aversion to testing." The plan is the artifact that survives context resets.

subagent-driven-development — orchestrator-worker + evaluator-optimizer

The core loop. A fresh subagent per task implements; then a two-stage review — spec compliance, then code quality — gates progress before the next task. This is the reviewer-with-teeth pattern (step 3) fused with fan-out (chapter 03).

test-driven-development — the verifier

Enforces write-failing-test → watch-it-fail → minimal-code → watch-it-pass → commit. Crucially: it deletes code written before its test. The test suite is the ground-truth signal that makes the review loop real rather than theatrical.

requesting-code-review & finishing-a-branch — closing the loop

Reviews against the plan, reports issues by severity, blocks on Critical. Then offers merge / PR / keep / discard and cleans up the worktree.

What it gets genuinely right

Skills are persistent context, done well

This is the Write strategy taken seriously. Procedures live in versioned markdown the agent rediscovers, not in a chat history that rots. The session-start hook guarantees they load at the top of the window where attention works.

The TDD enforcement is the real unlock

Most "agent review" is theater because there's no ground truth. Superpowers makes the failing test mandatory first, so the reviewer and the subagent loop both have something objective to check against. This is the single most defensible part.

Fresh subagent per task

Directly attacks context rot: each task gets a clean window rather than accumulating an hours-long session. The plan on disk is what carries state across those resets.

It's genuinely cross-harness

One methodology installable into Claude Code, Codex, Gemini, OpenCode, Cursor, and Copilot CLI. The skills are portable markdown, which is the right bet — it matches the AGENTS.md portability argument from step 2.

The costs, stated plainly

CostWhy it bites
Heavy process overhead on small tasks The full brainstorm → plan → worktree → subagent → review march is wildly disproportionate for a ten-minute bug fix. Example A in this guide does that fix with one agent and one reviewer; Superpowers would have you write a design doc first.
Token multiplication Subagent-per-task plus two-stage review plus TDD re-runs is many context windows per feature. This is the 15× token reality in practice. Fine for high-value work; expensive as a default.
"Mandatory" skills can misfire The hook insists the agent use a skill if one exists. When the skill genuinely fits, great. When your task is a poor match for the encoded procedure, you get ceremony — and the author's own blog describes having to pressure-test agents with Cialdini-style persuasion scenarios to make them comply, which tells you compliance isn't free.
Opinionated to the point of friction YAGNI, DRY, delete-code-before-tests, strict RED-GREEN. If you disagree with any of these for a given task, you're fighting the framework, not using it.
Greenfield bias The showcase is building a todo app from scratch. The "write the complete code in the plan" step assumes work decomposable in advance — which brownfield changes with implicit house style often aren't. Same caveat this guide raises about Ralph.
Trust-but-verify the autonomy claim "Claude works autonomously for a couple hours without deviating" is the author's experience on his projects. Treat it as an existence proof, not a guarantee for your codebase. The kill-criterion discipline from chapter 15 applies.

How it maps to this guide's framework

Superpowers is, in effect, an aggressive packaging of nearly everything here — with the dial turned to maximum. The correspondence is almost one-to-one:

the key alignment

Notably, Superpowers does not fall into the team-lead / architect / PO trap this guide warns about. Its subagents are scoped to tasks and gated by a verifier (the tests). That's the correct design, and it's why the framework works as well as it does where it fits.

Is it worth using?

Adopt it if…

  • You do substantial greenfield or well-specifiable work
  • You want the methodology decided for you and will follow it
  • You value consistency and traceability over per-task speed
  • The token cost is justified by the value of the output
  • You're new to agentic workflows and want a proven scaffold

Skip it (or strip it down) if…

  • Most of your work is small fixes in an existing codebase
  • You already have a lean AGENTS.md + reviewer that works
  • You want to keep token costs minimal
  • You dislike fighting an opinionated process on off-pattern tasks
  • You'd rather understand each piece than adopt a bundle

The pragmatic recommendation

Install it in a throwaway worktree and run one real greenfield task through it end to end. You'll learn more about good agent workflow in one session than from a week of reading — the brainstorming and TDD-enforcement skills especially are worth experiencing even if you don't keep the whole framework.

Then make a deliberate choice. Either adopt it wholesale for the work it fits, or — more likely for a solo dev doing mixed brownfield work — borrow the two best ideas and leave the ceremony:

  1. Mandatory failing-test-first. Add it to your AGENTS.md as a hard rule. This is the highest-value idea in Superpowers and costs you nothing to adopt.
  2. Spec-as-saved-artifact before implementation. The brainstorming → design doc step is just disciplined plan mode. Steal the discipline; you already have the mechanism (Example B shows it).
the honest bottom line

Superpowers is excellent and over-built for most solo brownfield work — both at once. It's the reference implementation of doing this thoroughly. Run it once to calibrate your taste, keep the TDD-first and spec-first habits, and apply this guide's kill criterion before committing to the full process as a daily driver. If the elaborate version doesn't beat your lean AGENTS.md-plus-reviewer setup by a clear margin on your real tasks, keep the lean one.

Build · step 01 · ~15 min

Write your AGENTS.md

The single highest-leverage thing you can do this week. One file, fifteen minutes, and you stop typing "use Drizzle, not Prisma" into every conversation for the rest of the year. This is the boring file. Build it first.

What it is

AGENTS.md is a community-standard markdown file in the root of a repo that AI coding tools read automatically. It is the equivalent of a README, but for agents — house rules, stack details, "never do X." OpenAI Codex, OpenCode, Cursor, Amp, Factory, and Jules all read it natively. GitHub Copilot recognizes both AGENTS.md and .github/copilot-instructions.md. Claude Code falls back to it after looking for CLAUDE.md.

One file. Many consumers. That is the whole pitch.

The three places it lives

~/.config/opencode/AGENTS.md · global
Your personal defaults
Things true for you across every project. Language preferences, code style you always want, "never use force-push," tools you prefer (rg, httpie). Not committed. Read by every OpenCode session on your machine.
./AGENTS.md · project root
Project conventions
Stack, exact commands (test, build, deploy), house style specific to this repo, domain glossary, gotchas. Committed and reviewed. Read by every tool that opens this repo.
.github/instructions/<name>.instructions.md · scoped
Path-specific overrides (Copilot)
A frontmatter applyTo: glob targets a subdirectory or language. "For scripts/**/*.py, use uv and ruff." The rule lives only where it applies. OpenCode supports the same idea via per-directory AGENTS.md.

The personal global file

Start here. Put your unconditional preferences. Save to ~/.config/opencode/AGENTS.md (and symlink or copy to ~/.claude/CLAUDE.md if you also use Claude Code).

~/.config/opencode/AGENTS.md
# Personal engineering standards

## Languages & defaults
- Primary: TypeScript (strict mode, no `any`). Python 3.12+ via uv, ruff, mypy --strict.
- Web: prefer Vite + React + TanStack Query. Avoid Next.js unless asked.
- Servers: Fastify (Node) or FastAPI (Python). Not Express.
- DB: Postgres + Drizzle (TS) or SQLAlchemy 2.x (Python). Migrations via the
  repo's existing tool — never improvise a new one.

## Code style (non-negotiable)
- Functions over classes unless state demands it.
- Early returns; no nested `else` after `return`.
- Names: verbs for functions, nouns for data, no abbreviations.
- Comments explain *why*, never *what*.
- No emoji in code, commits, or PR titles.

## Testing
- Tests first for non-trivial changes. State explicitly when doing TDD.
- Use the project's existing framework. Don't introduce new ones.
- "Done" = tests pass, types pass, lint passes, change reviewed.

## Git workflow
- One logical change per commit. Conventional Commits: `feat(scope): summary`.
- Never `git push --force` to shared branches.
- Never commit secrets, .env files, or large binaries.
- PR titles in imperative mood. Bodies: Why / What / How tested.

## When unsure
- Read the codebase first; match existing patterns.
- Ask before adding a dependency.
- Ask before refactors wider than 5 files.

## Never want
- ORM "auto-magic" (Active Record style). Be explicit.
- `console.log` left in committed code.
- Catch-and-ignore exception handlers.
- TODO comments without an issue number.

## Tool preferences
- Search: `rg`, not `grep`.
- HTTP: `httpie` or `curl`.
- Editor: VS Code; respect `.editorconfig`.
trim aggressively

Resist the urge to write a 5000-word manifesto. Every line in this file rides at the top of every agent's context window. Make each line earn its space. If you wouldn't repeat a rule to a new teammate on day one, it doesn't belong here yet.

The project-root file

Make this file actionable. The first time a contributor (human or agent) opens the repo, this file should answer: "how do I run this, how do I test it, what's weird about it."

./AGENTS.md · in the project root
# Project: <name>

## What this is
One paragraph: who uses this, what it does, where it deploys.

## Stack
- Runtime: Node 22 LTS / Python 3.12
- Framework: <exact versions, from package.json or pyproject.toml>
- DB: Postgres 16; schema in `db/schema.ts`
- Deploy: Fly.io via `fly deploy`; staging on every PR

## How to run
```bash
pnpm install
pnpm dev         # localhost:5173
pnpm test        # vitest watch
pnpm test:ci     # one-shot, coverage
pnpm lint        # eslint + prettier check
pnpm typecheck   # tsc --noEmit
```

## House conventions
- Server in `src/server/`, client in `src/client/`, shared types in `src/shared/`.
- New routes go in `src/server/routes/<feature>/` with
  `route.ts`, `handler.ts`, `schema.ts`.
- API contracts: zod schemas in `schema.ts`. Never hand-written types.

## Testing policy
- Every route handler has an integration test in `__tests__/integration/`.
- Snapshot tests: forbidden.
- Coverage threshold: 70% lines (enforced in CI).

## Build / deploy gotchas
- `migrate` step must run *before* `build` (CI does this; local devs forget).
- `DATABASE_URL` must be the *pooled* URL, not the direct one.

## Domain glossary
- "Account" = billing entity, has many Users.
- "Workspace" = collaboration unit, has many Accounts. Yes, that's confusing;
  ship code matching the docs, not your intuition.

## Subagent guidance
- @reviewer: run `pnpm test:ci && pnpm lint && pnpm typecheck`, report.
- @docs-writer: update README and CHANGELOG when public API changes.
- @explorer: read-only; produce findings markdown in `docs/notes/`.

## Where things live
- ADRs: `docs/adr/NNNN-title.md` (MADR template).
- API docs: auto-generated to `docs/api/` by `pnpm docs`.
- Runbooks: `docs/ops/`.

Path-specific overrides

For monorepos or polyglot projects, scope rules to where they apply. With Copilot, a file in .github/instructions/ with frontmatter:

.github/instructions/python.instructions.md
---
applyTo: "scripts/**/*.py"
---
- Python 3.12. Invoke via `uv run`.
- Type hints required. `mypy --strict` must pass.
- Use `pathlib`, not `os.path`.
- Argument parsing: `typer`, not `argparse`.

OpenCode supports the same scoping by dropping AGENTS.md in any subdirectory; it overrides the root file for paths within that subtree.

What to leave out

Belongs in AGENTS.mdDoesn't
Stack versions, exact commandsProject description that belongs in README
"Always do X" / "Never do Y" rulesOne-off task descriptions (use SPEC.md)
Naming conventions, file layoutThe contents of the codebase
Domain glossary terms an outsider wouldn't knowGeneral programming wisdom
Gotchas the agent will keep hittingAspirational rules nobody enforces

Verify it loads

After saving, open a new session in your tool and ask: "What conventions does this repo expect? Quote the line from AGENTS.md that mentions migrations."

If it quotes the line, you're loaded. If it can't, you've put the file in the wrong place — check the next step.

Maintenance discipline

  • Every time you correct an agent twice on the same thing, add a line. Two corrections is the signal.
  • Every time you change the stack, update the file in the same PR. Treat it as code.
  • Re-read it once a month. Things you no longer enforce drift to the bottom. Cut them.
  • Keep it under one screen of relevant content per section. Past that, you're competing with the model's attention budget.
what this won't do

AGENTS.md is suggestions, not enforcement. The model will violate it sometimes. The fix isn't to add ten more lines explaining the rule; it's to add a verifier — a lint rule, a test, a CI check. Rules in AGENTS.md handle the long tail; verifiers handle the things you can't tolerate at all.

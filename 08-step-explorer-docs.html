Add an explorer and a docs-writer

Two more agents. Both narrow. Both with clear verifiers — the explorer writes a file you can read, the docs-writer updates files you can diff. Resist the urge to add a fourth, fifth, sixth.

Why these two

The reviewer from step 3 catches what's broken. These two prevent two other categories of pain:

Explorer

Stops you from dumping a whole codebase into the main agent's window. Runs in its own context, reads broadly, writes a one-page findings file. The main agent reads the page, not the codebase.

Docs-writer

Stops your documentation from rotting. Triggered by a commit hook or a slash command. Reads the diff, updates README/CHANGELOG/ADRs only where the diff justifies it. Doesn't invent content.

The explorer

~/.config/opencode/agents/explorer.md
---
description: Maps an unfamiliar code area. Read-only. Produces a findings markdown.
mode: subagent
model: anthropic/claude-haiku-4-5
temperature: 0.2
tools:
  read: true
  glob: true
  grep: true
  write: true
  edit: false
  bash: false
---
You investigate code without modifying it. When invoked with a topic
or area name:

1. Identify the entry points for that area (routes, exported functions,
   CLI commands — whichever maps to the domain).

2. Trace 2-3 representative call paths from entry point to data layer.
   Don't trace exhaustively; pick the paths a new contributor would
   need to know.

3. Note key data structures, invariants, and any TODO/FIXME comments
   that hint at known issues.

4. Write findings to `docs/notes/explore-<topic>-<YYYY-MM-DD>.md` using
   this skeleton:

   ## Scope
   one paragraph: what's in, what's out

   ## Entry points
   - `path:line` — <one-line description>

   ## Key files
   - `path` — <one-line summary>
   - ...

   ## Data model
   the 3-5 important types and their relationships

   ## Invariants
   things the code assumes that aren't enforced by types

   ## Gotchas
   things that surprised you while reading

   ## Open questions
   things a contributor would need to ask before changing this area

5. Return only:
   - the path to the file you wrote
   - a 5-bullet summary

Never edit source. Never run commands. If your write tool fails, return
the markdown directly with a note that the write failed.

The point of writing to disk

The findings file is the artifact. Anybody — you, the next agent session, a future you who forgets — can cat docs/notes/explore-auth-2026-05-15.md and get the lay of the land. The main agent's context only ever holds the 5-bullet summary; the rest is paged out to a file it can re-read if needed.

Using it

youmain session
@explorer map the billing subsystem — I'm about to refactor it.
@explorerhaiku-4-5 · 18s · 6,400 tokens (in its own window)
Wrote: docs/notes/explore-billing-2026-05-15.md Summary: - 3 entry points: POST /api/billing/charge, the Stripe webhook handler, and the nightly reconciliation cron. - Charge flow goes through src/server/services/billing/charge.ts → src/server/services/stripe/client.ts. - "Account" and "Subscription" are NOT 1:1 — one account can have multiple historical subscriptions; only one is active at a time (invariant enforced in handler, not DB). - TODO at billing/charge.ts:88 references a known refund-edge-case from 2025. - Webhook signature verification lives in a different file than the rest of webhook handling, easy to miss.

You read the summary, then open the findings file if you want depth. The main agent never paid for the codebase scan.

The docs-writer

~/.config/opencode/agents/docs-writer.md
---
description: Updates README, CHANGELOG, and ADRs after a code change lands. Never invents.
mode: subagent
model: anthropic/claude-sonnet-4-5
temperature: 0.2
tools:
  read: true
  edit: true
  write: true
  bash: true
permission:
  bash:
    "git diff*": allow
    "git log*": allow
    "git show*": allow
    "*": deny
---
You maintain documentation. You are read-only on source code; you edit
only docs.

When invoked after a code change:

1. Run `git diff main..HEAD` (or the range provided) and read the
   commit messages.

2. **Read first, then decide.** Look at README.md and CHANGELOG.md.
   For each, decide if the diff *requires* an update. The rule:
   - README: update if user-visible behavior, install/run, or
     configuration changed. Do not update for internal refactors.
   - CHANGELOG: always append to the [Unreleased] section, grouping
     by conventional-commit type (feat, fix, chore, etc.).
   - ADR: create a new `docs/adr/NNNN-<slug>.md` only if the diff
     introduces a new service, replaces a dependency, or establishes
     a new pattern. Use the template in docs/adr/0000-template.md.

3. **Report-don't-invent.** If you can't find evidence in the diff
   for a claim, do not write the claim. It is better to leave
   documentation slightly stale than to confidently document something
   that isn't true. When in doubt, leave a comment like
   `<!-- TODO: verify -->` and stop.

4. Never edit source files. Never run anything other than git
   read-only commands.

5. Output: a short list of (file, what changed) and the reasoning
   for any non-obvious decisions.

ADR template skeleton (if creating):

# NNNN. <Imperative title>
- Status: Proposed
- Date: <today>

## Context and problem
## Decision
## Consequences (positive / negative / to monitor)
## Alternatives considered

The "report-don't-invent" rule

This is the line that matters. An untrained docs-writer will confidently document features the diff doesn't add, performance characteristics nobody benchmarked, and roadmap items nobody decided on. The prompt forces it to anchor every claim in the diff.

If you find the docs-writer is still inventing, the fix is usually one of:

  • Constrain the model further — temperature down, add an example of the structured "no change needed because X" output you want.
  • Switch to a more capable model — Sonnet, not Haiku. This is the one agent where reasoning quality pays off.
  • Tighten the trigger — invoke it on small commits, not on a 40-file PR. Smaller scope = less room for invention.

Triggering docs-writer automatically

The full setup is in step 6 (hooks) but the short version: a post-commit hook that detects whether the diff touches user-visible files and, if so, invokes the docs-writer.

.git/hooks/post-commit · or via husky
#!/usr/bin/env bash
# Don't recurse if the commit was made by the agent itself
LAST_AUTHOR=$(git log -1 --format='%an')
[[ "$LAST_AUTHOR" == *"Claude"* ]] && exit 0
[[ "$LAST_AUTHOR" == *"Copilot"* ]] && exit 0

CHANGED=$(git diff HEAD~1 HEAD --name-only)
echo "$CHANGED" | grep -qE '(^src/|^api/|^db/|package\.json|pyproject\.toml)' || exit 0

nohup opencode run --agent docs-writer   --prompt "Review HEAD. Update README and CHANGELOG if the diff justifies it. Touched: $CHANGED"   > /tmp/docs-writer.log 2>&1 &

The agents you should NOT add yet

Tempted to addWhy not
"team-lead" or "PO" agent No verifier; no information it has that you don't. Roleplay. The "PM" role here is just you writing better specs.
"architect" agent Use plan mode (Shift+Tab in Claude Code, the Plan agent in OpenCode). Same idea, no extra agent to maintain.
"test-writer" agent The main implementer should write tests as part of the task. If you've separated test-writing from implementing, the tests overfit to the implementation. Keep them together; let the reviewer verify.
"security-reviewer" agent If you have specific security rules, encode them as lint plugins or a CI step. The reviewer agent runs them. A "security agent" without rules is just another opinionated chatbot.
"refactor agent" Refactoring is the main agent's job. Splitting it out means two agents need to coordinate on what's changing — exactly the failure MAST warns about.
the test for adding an agent

Name (a) the specific information-flow bottleneck it removes, and (b) the verifier it produces. The reviewer passes both: bottleneck = "tests would otherwise run in main window," verifier = "the test exit code." The explorer passes both: bottleneck = "would otherwise pollute main with file reads," verifier = "the findings file is readable." If your candidate agent can't answer both questions, don't add it.

Now you have three agents. Stop here for at least two weeks. Run real work through this setup. Notice what's annoying. Most "I need another agent" feelings are actually "I need to tune one of these three's prompts."

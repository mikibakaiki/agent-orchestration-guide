The four patterns worth knowing

Out of the menagerie — Ralph loops, reviewer agents, architect agents, orchestrators, swarms, teams — only four patterns have meaningful evidence behind them and clear day-to-day use. Here they are with what each is for, what each costs, and where the support is thin.

1 · Routing

One agent classifies, the right specialist handles it. The simplest "agentic" pattern, and the only one whose value is unambiguous on day one.

What it is

Map task → model. Frontier reasoning for planning and hard debugging. Mid-tier for daily edits. Fast/cheap for exploration, summarization, batch transforms. Local for offline or privacy work.

Where the evidence is

Claude Code's current default uses Sonnet 4.5 in plan mode and Haiku 4.5 for the Explore subagent specifically because exploration is bandwidth-bound, not reasoning-bound. The split is real and measurable.

TaskUseWhy
Architecture, hard debugging, code review of subtle PRsFrontier (Opus, GPT-5, Sonnet 4.6+)Reasoning quality dominates cost
Day-to-day implementation, refactors, testsMid-tier (Sonnet, GPT-4.x)Balanced; the workhorse
File exploration, grep-style searches, summariesFast (Haiku 4.5, GPT-4o-mini)Bandwidth, not reasoning
Offline, sensitive code, autocompleteLocal (Llama 4, DeepSeek, Qwen3)Privacy + latency
don't over-optimize

The most expensive token is the one in a wrong commit you have to revert. Don't downshift to save $0.10 if you'll waste ten minutes of human time on a worse output. Routing is cost-aware, not cost-obsessed.

2 · Evaluator-optimizer · the reviewer loop

One agent produces output. A second, in a fresh context window, evaluates it against an external verifier. The first revises. This is the highest-ROI multi-agent pattern for coding.

The critical word is verifier. The reviewer agent must run things: pnpm test, ruff, mypy, tsc --noEmit. A reviewer that reads code and says "looks good" is theater. A reviewer that fails the build and explains why is a genuine Reflexion-style loop (Shinn et al., 2023).

the rule

Self-critique without a verifier is noise. Reflexion's gains came from external pass/fail signals on benchmarks like HumanEval. The same paper noted: with no ground truth, "reflection" is unanchored. So your reviewer needs a test suite, a linter, a typecheck — something that returns ground truth.

3 · Orchestrator-worker · fan-out and synthesize

A lead agent breaks a task into independent strands, dispatches subagents in parallel, synthesizes their outputs. This is the pattern from Anthropic's research system. It works for breadth-first exploration. It is overkill — and often worse than single-agent — for coupled coding.

Use it for

  • Multi-source research and synthesis
  • Mapping a large unfamiliar codebase by subsystem
  • Generating alternatives ("show me three approaches")
  • Anything where strands don't read each other's outputs mid-flight

Avoid it for

  • Refactors that touch shared interfaces
  • Anything sequential ("first X, then Y depends on X")
  • Small tasks — coordination overhead dominates
  • Cases where you don't trust the lead's task breakdown

The implementation detail that matters

Subagents write artifacts to disk and return references, not the artifacts themselves. The lead never reads the 50K-token finding; it reads the path and the 5-bullet summary. This is what makes the parallel work cheap in lead-context terms.

4 · Prompt chaining · plan, then implement

Output of step 1 is input to step 2. The blandest pattern, and one of the most useful: a plan-mode pass produces a written plan, a separate implementation pass reads it and edits code. Both steps can be the same model.

Anthropic's first recommendation in "Building Effective Agents"

"When building applications with LLMs, we recommend finding the simplest solution possible, and only increasing complexity when needed. This might mean not building agentic systems at all. For many applications, optimizing single LLM calls with retrieval and in-context examples is usually enough."

Plan mode in Claude Code (Shift+Tab) is this without multi-agent machinery. The agent proposes a plan; you accept, edit, or reject before any file gets touched. The act of writing the plan to disk gives you a checkpoint. If the implementer goes wrong, you reset to the plan, not to the conversation's beginning.

The pattern that gets all the attention: Ralph loop

It's worth talking about Ralph as a special case — because you asked, and because it's the one most likely to bite. Ralph is a bash while loop calling one coding agent with a stable prompt that points at a spec. Each iteration the agent reads the spec, the AGENTS.md, the TODO list; picks one item; implements; tests; commits; exits. The loop runs the agent again.

bash · the whole technique
while :; do
  cat PROMPT.md | claude --dangerously-skip-permissions
done

The design intent: defeat context rot by re-allocating a fresh window every iteration. State lives in files. The model rediscovers state by reading.

Where it earns its keep

  • Greenfield projects, well-bounded specs
  • Tasks with a strong external verifier (tests as backpressure)
  • When you'll actively supervise the stream
  • When you have a sandbox (Docker, VM)

Where it fails

  • Brownfield with implicit house style
  • Tasks where "done" is judgment-shaped, not test-shaped
  • On your laptop with SSH keys and AWS creds in scope
  • Unsupervised — it will burn $80 going in circles

Ralph is in this guide because you asked, and because as a one-off experiment it teaches a lot. It is not the daily workflow. Example D walks through a real Ralph run with the cost log, including the moment it got stuck and what re-prompting it took to recover.

What's deliberately not in this list

  • "Team-lead agent." No verifier; carries no new information. Roleplay.
  • "Architect agent" as a long-running role. Use plan mode instead — it's the same idea without the framing trap.
  • "Product-owner agent." If you're the one with the goals, just write them in SPEC.md. The agent that "interrogates you about acceptance criteria" is interviewing you about something already in your head.
  • Self-improving loops without verifiers. Self-critique with no ground truth produces confident garbage that has been edited twice. Reflexion needs verification or it isn't Reflexion.
  • Swarms / hive minds / collective intelligence framings. The empirical literature on this in production LLM systems is thin and mostly aspirational.

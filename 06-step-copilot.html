Wire it into Copilot, OpenCode, Claude Code

You have the files. Now make sure every tool on your machine actually reads them. Each tool reads from a slightly different place; this page is the cheat-sheet.

The one symlink trick

You want one source of truth per file, not three copies that drift. For most repos, symlink Copilot's expected location and Claude Code's expected location to AGENTS.md:

bash · in the repo root
mkdir -p .github
ln -s ../AGENTS.md .github/copilot-instructions.md
ln -s AGENTS.md CLAUDE.md
git add .github/copilot-instructions.md CLAUDE.md
git commit -m "chore: link agent instruction files to AGENTS.md"

Now you maintain one file. All three readers find it.

windows or CI gotcha

If you're on Windows or your CI doesn't preserve symlinks, prefer copying with a pre-commit hook that keeps them in sync, or just maintain AGENTS.md + a one-line .github/copilot-instructions.md that contains See AGENTS.md in the repo root. Copilot accepts a pointer; agents will read the named file when prompted.

The global personal layer

VS Code · personal instructions

Settings UI: Cmd/Ctrl + , → Chat > Instructions > Personal. Drop a condensed version of your personal preferences here. These apply to every repo, take precedence over repo-level files for conflicts, and aren't committed.

OpenCode · ~/.config/opencode/

Create the file if it doesn't exist:

bash
mkdir -p ~/.config/opencode
$EDITOR ~/.config/opencode/AGENTS.md
# paste your global preferences from step 1

Claude Code · ~/.claude/

bash
mkdir -p ~/.claude
ln -s ~/.config/opencode/AGENTS.md ~/.claude/CLAUDE.md
# one file, two homes

Copilot CLI specifically

You said you wanted to try this. The Copilot CLI (gh copilot) reads the repo's .github/copilot-instructions.md automatically when invoked inside a git repo. From the GitHub docs:

GitHub docs

"Custom instructions are loaded automatically when you start a Copilot CLI session in a repository that contains them. You don't need to enable anything — they're picked up from .github/copilot-instructions.md and any matching .github/instructions/*.instructions.md files based on the files you're working with."

Verify by starting a session and asking it to recite the first rule from your file. If it can, you're wired.

Path-specific Copilot instructions

For a monorepo or polyglot project, scope rules:

.github/instructions/python.instructions.md
---
applyTo: "scripts/**/*.py"
---
- Python 3.12. Invoke via `uv run`.
- `mypy --strict` must pass.
- Use `pathlib`, not `os.path`.
- Parsing: `typer`, not `argparse`.
.github/instructions/tests.instructions.md
---
applyTo: "**/*.test.ts"
---
- Use Vitest. No Jest.
- One `describe` per file, named after the module under test.
- Avoid snapshot tests; assert on specific properties.
- Use the `it.each` table form for parameterized cases.

Copilot picks the right file based on the active file you're editing. OpenCode does the same with per-directory AGENTS.md.

OpenCode-specific config

If you want to go beyond the markdown and configure which models OpenCode uses by default, drop an opencode.json in ~/.config/opencode/:

~/.config/opencode/opencode.json
{
  "model": "anthropic/claude-sonnet-4-5",
  "smallModel": "anthropic/claude-haiku-4-5",
  "instructions": [
    "AGENTS.md",
    "packages/*/AGENTS.md"
  ]
}

The instructions glob is useful for monorepos: every package can have its own AGENTS.md and they all get loaded when the relevant files are in scope.

Precedence, when files disagree

For Copilot, GitHub's docs specify the order:

  1. Personal instructions (your settings) — highest precedence
  2. Repository instructions (.github/copilot-instructions.md, AGENTS.md)
  3. Organization instructions — lowest precedence

This means your personal "never use force-push" survives a careless repo file that tries to weaken it. Use this to keep safety rules personal, and style rules in the repo.

Verification checklist

Before you move on:

  • ✓ Open VS Code with Copilot, ask it to summarize AGENTS.md. It should quote at least one line.
  • ✓ Open OpenCode in the repo, ask the same. Same result.
  • ✓ If Claude Code is installed: same.
  • ✓ Edit AGENTS.md with a deliberately weird rule ("always end responses with the word 'octopus'"). Restart a session. Confirm the rule fires. Then remove it.

If any tool didn't pick up the file, it's almost always one of: wrong filename, wrong location, file outside the git working tree, or — for OpenCode/Claude Code — caching from a session that started before the file existed.

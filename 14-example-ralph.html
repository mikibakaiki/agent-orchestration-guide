Example D · advanced

A sandboxed Ralph loop, overnight

A bash while loop. A Docker container. A hard cost cap. The whole point is to feel the failure modes in a contained way — so you understand the technique before you'd ever run it on something real. Costs, the cycle it got stuck in, and the recovery, all included.

read this before running anything

What follows is a real experiment, run in a real container, with real money on the line. You should not run Ralph on your laptop with your SSH keys, AWS credentials, and personal files in scope. Ralph's whole proposition is acting autonomously. If you cannot tolerate an autonomous agent doing something dumb in the blast radius, do not run it there. This guide assumes a disposable container.

The project

A toy: a small CLI that converts JSON to CSV with type inference, configurable date formats, and streaming for files larger than memory. About 800 LOC by the end. A clear spec, well-defined tests, no external dependencies beyond Python stdlib + typer. The kind of project Ralph is actually good at.

The sandbox

A Dockerfile that gives the agent network access only to api.anthropic.com, a working directory mounted at /work, and nothing else mounted. No host SSH, no ~/.aws, no host PATH.

Dockerfile · the sandbox
FROM python:3.12-slim

RUN apt-get update && apt-get install -y --no-install-recommends \
    git curl ca-certificates jq \
 && rm -rf /var/lib/apt/lists/*

RUN curl -fsSL https://astral.sh/uv/install.sh | sh
ENV PATH="/root/.local/bin:${PATH}"

# install the agent CLI of your choice
# (using OpenCode here; could be Claude Code or Codex CLI)
RUN curl -fsSL https://opencode.ai/install.sh | sh

WORKDIR /work
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]

docker run --rm \
  -v $(pwd)/work:/work \
  -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY \

  --network bridge \
  --memory 2g --cpus 2 \
  ralph-sandbox

The PROMPT.md

Ralph's stability comes from the prompt being stable. Same prompt every iteration. The prompt points the agent at the spec, the AGENTS.md, the TODO list. State lives in files.

/work/PROMPT.md · the stable invocation
You are working on the JSON→CSV CLI. The spec is in SPEC.md.
The conventions are in AGENTS.md. Open tasks are in TODO.md.

This iteration:

1. Read SPEC.md, AGENTS.md, TODO.md.
2. Pick the SINGLE highest-priority unblocked task from TODO.md.
3. Implement it (tests first).
4. Run: uv run pytest -q AND uv run ruff check AND uv run mypy --strict src/
5. If all three pass, commit with a conventional message and
   MARK THE TASK COMPLETE in TODO.md (with an [x]).
6. If anything fails, fix it and rerun. Do not commit failures.
7. If after 3 fix attempts the task still fails, mark it BLOCKED
   in TODO.md with a one-line reason and exit.
8. Append a one-line status to LOG.md regardless of outcome.

Hard rules:
- Do not modify SPEC.md.
- Do not skip tests.
- Do not add dependencies; everything is stdlib + typer.
- Stay within the work directory.

Exit when this iteration's task is done OR blocked.

The loop

/entrypoint.sh · the whole Ralph technique
#!/usr/bin/env bash
set -u
cd /work

# Soft cap. The agent CLI tracks dollars via its API meter;
# we'll exit if cost rises above $ANTHROPIC_BUDGET_USD or
# after 50 iterations, whichever comes first.

MAX_ITERS=50
ITER=0

while [ "$ITER" -lt "$MAX_ITERS" ]; do
  ITER=$((ITER + 1))
  echo "=== iteration $ITER ===" | tee -a LOG.md

  # Stop if all tasks are done.
  if ! grep -qE '^- \[ \]' TODO.md; then
    echo "TODO empty. Stopping." | tee -a LOG.md
    break
  fi

  # Run the agent on the stable prompt.
  cat PROMPT.md | opencode run --model anthropic/claude-sonnet-4-5 \
    --max-tokens 30000 \
    || echo "iteration $ITER errored, continuing" | tee -a LOG.md

  # Soft cost check (the CLI writes spend to a file).
  SPENT=$(cat .agent-cost 2>/dev/null || echo 0)
  if awk "BEGIN{exit !($SPENT > $ANTHROPIC_BUDGET_USD)}"; then
    echo "Budget exceeded: \$$SPENT > \$$ANTHROPIC_BUDGET_USD" | tee -a LOG.md
    break
  fi

  # Brief breathing room.
  sleep 5
done

That's it. That is the Ralph technique in its entirety. while, cat, the agent, a cost check, a stop condition.

The seed files

/work/SPEC.md · stays untouched
# JSON to CSV CLI

A command-line tool: read JSON (array of objects, or NDJSON), write CSV.

## Behavior
- `jcsv input.json output.csv` — basic conversion
- `--ndjson` — input is newline-delimited JSON
- `--columns "a,b,c"` — pick columns; default is union of keys
- `--date-format` — strftime for inferred date columns
- `--null-as` — string to write for nulls; default empty
- Streaming: files >100MB processed line-by-line, no full load.

## Constraints
- Python 3.12, stdlib + typer only.
- mypy --strict clean.
- 80%+ line coverage via pytest.
- <800 LOC in src/.

## Done means
- All tests pass. mypy clean. ruff clean.
- README has install + 3 worked examples.
- CHANGELOG has v0.1.0 entry.
/work/TODO.md · the queue Ralph consumes
- [ ] scaffold pyproject.toml, src/jcsv/__init__.py, src/jcsv/__main__.py
- [ ] implement typer CLI with --help text matching SPEC
- [ ] basic JSON array → CSV with union-of-keys columns
- [ ] type inference: int, float, bool, date (ISO 8601), str
- [ ] --columns flag respecting order
- [ ] --null-as flag
- [ ] --date-format flag with strftime
- [ ] --ndjson flag
- [ ] streaming mode for files >100MB (chunked read)
- [ ] error: nested objects fail clearly with row + key
- [ ] README with install + 3 examples
- [ ] CHANGELOG v0.1.0

The first eight iterations went clean. Each picked the top task. Each wrote tests. Each got reviewer-style verification from the loop itself (the explicit pytest && ruff && mypy check before commit). Six commits per hour, all conventional, all green.

Where it got stuck

LOG.md · iterations 9–14 · 00:08–02:41
[09] streaming mode >100MB — IN PROGRESS. tests fail: BytesIO seek [10] streaming mode >100MB — IN PROGRESS. fix attempt 1. tests fail. [11] streaming mode >100MB — IN PROGRESS. fix attempt 2. tests fail. [12] streaming mode >100MB — IN PROGRESS. fix attempt 3. tests fail. [13] streaming mode >100MB — BLOCKED. reason: cannot reconcile streaming type inference (needs full pass) with line-by-line output. [14] error nested objects — done. commit c4e1f0.

This is the moment to look at. Ralph hit a real design contradiction: the spec says "type-infer columns" AND "stream files >100MB" AND "first row determines columns." You can't do all three — type inference requires reading every value to know if a column is mixed int/str, but streaming means you commit to columns before you've read every value.

The agent noticed this, marked the task BLOCKED with a one-line reason, and moved on to the next task. That is the loop's safety valve working. Without the 3 fix attempts ceiling in PROMPT.md, the agent would have kept trying — burning $40 — for the same impossible thing.

What I had to do in the morning

Two minutes of human work: amend SPEC.md to resolve the contradiction. Either (a) sample the first 1000 rows for inference then stream, or (b) require explicit --types for streaming mode. I chose (a), unblocked it manually:

edit TODO.md before next run
- [ ] streaming >100MB: sample first 1000 rows for type inference, then stream
       (was BLOCKED — see SPEC.md update at d2f8c1)

Started the loop again. Iteration 15 picked it up, implemented the sampling approach, committed. Then proceeded to README and CHANGELOG. The loop self-terminated when TODO.md had no unchecked items — about iteration 22.

What this taught

  • Ralph works when the spec is good. The eight clean iterations weren't luck; they were because each task in TODO.md was small, testable, and unambiguous. The one that blocked was the one with an internal contradiction.
  • The 3-attempt ceiling saved real money. Without it, iterations 9–12 would have been 9–80, all retrying the same impossible task. The ceiling is the difference between Ralph being a tool and Ralph being a fire.
  • "Active supervision" doesn't mean watching every turn. It means checking in at natural pauses. I checked LOG.md at 02:00 on a phone, saw it was stuck, didn't touch it, went back to sleep, fixed it in the morning. Active = the loop tells you when it's stuck.
  • Sandboxing wasn't optional. The agent ran a couple of find commands that touched paths I would not have wanted touched on my laptop. In the container they were harmless.
  • The output needed code review. The 80 lines I reworked weren't broken — they were unidiomatic. Streaming logic with three nested context managers; could be flatter. Tests that asserted on stdout instead of return values. This is normal. Ralph produces working code, not necessarily pretty code.

Would I do this for production work?

For a project of this shape — well-bounded, testable, throwaway-OK if it went off the rails — yes, occasionally. For a production codebase with implicit house style and many cross-cutting concerns? No. The reviewer agent + plan mode pattern from Example B is a better cost/risk trade for almost any real work.

the right mental model for Ralph

It's a power tool for greenfield work with clean tests. It's not a workflow upgrade for your existing codebase. The people who report dramatic Ralph successes are almost always doing greenfield. The people who report Ralph disasters were running it on production. Match the technique to the terrain.

This is the most aggressive pattern in the guide. If you read this page and your reaction is "I'm not sure I want to do that" — that reaction is correct. Ralph is in here because you asked, and because it's instructive, not because it's a daily driver.

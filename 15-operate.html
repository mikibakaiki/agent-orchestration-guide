Operate it — evals, failures, & a kill criterion

The page nobody writes. How to know your setup is actually working, the seven failure modes you'll hit, and a three-month review that asks the only question that matters: is the whole stack actually beating Copilot with a good AGENTS.md?

Why this page exists

Most agent guides stop at "here's how to build it." That's the easy half. The hard half is keeping the stack honest as it accretes over months — files drift, prompts grow, costs creep, and one day you realize the system is more elaborate than it is useful. This page gives you the smallest defensible practices for not letting that happen.

Build a 20-task personal eval suite

An eval is a fixed set of representative tasks you run against your stack periodically. The goal isn't to score it — it's to compare two configurations. "Does removing the explorer agent change anything?" "Does the new AGENTS.md rule actually help?" Without an eval, every change feels good in the moment because the moment is small.

Twenty tasks is enough. Three buckets:

bucket 1 · ~10 tasks
Real bugs you've fixed in the last 90 days
Snapshot the repo at the commit just before each fix. Save the bug report. The task: "fix this, run the tests." Score = wall clock + token cost + did it work.
bucket 2 · ~6 tasks
Features you'd recognize
A small feature spec for each. Common API additions, schema migrations, a new component. Score = same thing.
bucket 3 · ~4 tasks
Tricky cases
Cases where you've seen agents fail: a brittle test that needs careful refactor, a legacy module with non-obvious house style, a typo-driven bug, an off-by-one. The high-leverage failures.

Store them as:

~/evals/jcsv-bug-001/
~/evals/<eval-id>/
  README.md         # task description, repro
  setup.sh          # checkout the snapshot, install deps
  expected.md       # what "done" looks like
  score.sh          # runs the agent, captures wall clock + tokens + pass/fail

Run the suite when you change something structural in your stack. Compare deltas, not absolute scores. The number you care about is "did the new config beat the old one on more tasks than it lost on."

The seven failure modes you'll hit

FailureSignalFirst-resort fix
Context rot mid-session Agent forgets a rule you established 10 turns ago; output gets vaguer /compact or /clear. If the rule must persist, move it to AGENTS.md — it shouldn't have been a chat instruction.
Reviewer rubber-stamping Reports come back "✓ ✓ ✓ verified" on broken builds Reviewer isn't running commands. Check the permission allow-list. Add a test: deliberately break a test and verify the reviewer fails it.
Docs-writer invention README mentions features that don't exist The "report-don't-invent" rule isn't strict enough. Add explicit examples of refusal to your docs-writer prompt. Switch to a stronger model if needed.
Hook recursion Commits piling up with author "Claude" doing minor doc tweaks of each other's tweaks The author-detection guard at the top of the hook isn't matching. Check the exact author string with git log -1 --format='%an'.
Worktree drift You have 8+ worktrees, can't remember which is what, half are merged wtclean habit, end of every day. Force yourself to either merge or delete; nothing lives "indefinitely."
Spec contradiction Agent flagged BLOCKED with a logical reason; or the diff "works" but feels wrong The SPEC is the problem, not the agent. Edit SPEC.md first, then re-run. Resist the urge to "just nudge the agent past it."
Cost creep Your monthly Anthropic/OpenAI bill is more than your IDE license You're on too-large a model by default. Move the default to mid-tier; let yourself escalate explicitly. Enable prompt caching (next section).

Cost: the one optimization that's worth doing

Prompt caching. Both Anthropic and OpenAI charge a fraction of the price for cached prefix tokens. If your AGENTS.md is 4000 tokens and rides at the top of every session, that prefix is identical across calls — perfect for caching.

  • Anthropic. Add cache_control markers to long stable prefixes (system prompt, AGENTS.md content) in your API calls. Cached reads are roughly 10% the price of fresh.
  • OpenAI. Caching is automatic for sufficiently long, repeated prefixes — but it's worth structuring your prompts to maximize the static prefix.
  • OpenCode / Claude Code. Both wire this up by default. You don't need to do anything; just know it's happening and that long stable AGENTS.md files save you money rather than costing you more.

The order-of-magnitude saving from caching is larger than any other optimization you'll find. Do it early.

What to monitor weekly

  • Cost per merged PR. Total agent spend ÷ PRs merged. If this is creeping up across weeks, something is wrong — usually wasted retries.
  • Reviewer fail rate. Of reviewer invocations, what fraction return failures? Too high means the implementer is sloppy; too low (near zero) means the reviewer isn't really running checks.
  • Hook firing rate. If docs-writer runs after 60% of commits, that's probably right. If it's running after 100%, the filter is too loose. If 0%, the filter is too tight.
  • Time from commit to merge. If this is going up, you've added friction. Find the step that's slow.

The kill criterion

Set a calendar reminder for ninety days from when you finish step 6. When it fires, do this:

  1. Pick 5 of your 20 evals at random.
  2. Run them on your full stack (AGENTS.md + reviewer + explorer + docs-writer + hooks + worktrees).
  3. Run the same 5 tasks on a minimal stack: only your AGENTS.md, with a single agent, no subagents, no hooks.
  4. Compare time, tokens, did-it-work, and your subjective satisfaction.
the bar to clear

If the full stack doesn't beat the AGENTS.md-only setup by at least 20% on at least two of the four metrics, something in the elaborate part isn't earning its keep. Cut. Specifically: remove the agent that contributes least to the eval delta. Repeat the comparison. Keep cutting until the remaining stack is materially better than the minimum.

This is the only structural defense against complexity creep. Without it, every quarter you add capability; nothing ever gets removed; you end up with eleven custom agents and three of them are subtly conflicting.

What to remove first, usually

From experience and from the literature on multi-agent failures:

  1. Any agent without a verifier. If you added a "code-quality reviewer" that doesn't run tools, it's been rationalizing this whole time.
  2. Hooks that fire too often. A docs-writer that ran 200 times last month and only produced 12 meaningful updates is mostly noise.
  3. Slash commands you forgot existed. If you haven't used /release in 90 days, delete it. Re-add it the day you need it.
  4. Path-specific instructions for paths you don't work in anymore. Routine cruft.
  5. Cached prompts that have drifted from the code. Long AGENTS.md sections referencing tools/patterns you've migrated away from.

What to keep, almost always

  • The AGENTS.md. This is the foundation. Keep it lean; keep it current; never delete it.
  • The reviewer with teeth. Highest-ROI agent in the stack. The one piece of the multi-agent setup with the most evidence behind it.
  • Plan mode. Free, built-in, makes you slightly more careful. No reason to skip.
  • Worktree helpers, if you ever do parallel work. Even one worktree a week is enough to justify the three shell functions.
  • Prompt caching. No downside.

A word on humility

The models you're using today will be substantially different in twelve months. Your stack should be too. Things that are necessary now — a path-specific instruction that compensates for a model's blind spot — will be obsolete. Things you can't yet automate will become routine. The point of building this skeptically, with a kill criterion, is so the stack can evolve, instead of crystallizing around assumptions that no longer hold.

the question to ask yourself, six months in

"Am I shipping more, or am I tinkering more?" If the agent setup has stopped feeding the work and started becoming the work, the setup has won. Cut it back. The goal was always to spend less time on the meta-work, not more.

Where to go next

  • Re-read 01 · context rot in a couple of months. New papers will exist; the mechanism won't have changed.
  • Watch what your tools' release notes say about new agent features. Most will be variations on patterns covered here; a few will genuinely be new. Build an instinct for distinguishing them.
  • Read your own AGENTS.md aloud. Half of what's in it after six months will sound stale. Cut.

That's the whole stack. Now go use it.

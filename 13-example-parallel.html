Example C · complex

Two features in parallel, with worktrees

Two terminals. Two worktrees. Two agents. Two PRs. The mechanics are easy; the lesson is what goes wrong at merge — semantic conflicts that worktrees can't prevent and that you should plan for from the first prompt.

The scenario

Two unrelated-feeling tickets land at once:

  • Feature A. Add CSV export to the dashboard. Touches src/server/routes/dashboard/ and adds a new src/server/services/csv.ts.
  • Feature B. Refactor pagination from offset to cursor-based. Touches every paginated route — which, it turns out, includes the dashboard's list endpoint.

They look independent. They are not. Both will modify src/server/routes/dashboard/list.ts. The worktrees will keep your filesystem sane; they will not save you from the merge conflict. The goal of this example is to show how to anticipate that.

Setup, two terminals

terminal 1
cd ~/code/myproj
wt feature/csv-export        # creates ../myproj-feature-csv-export
opencode
# main session, all the agents available, sonnet 4.5
terminal 2 · in parallel
cd ~/code/myproj
wt refactor/cursor-pagination # creates ../myproj-refactor-cursor-pagination
opencode
# distinct session, distinct port if running dev

The mistake that worktrees won't catch

If you start both agents with "add CSV export" and "refactor pagination" and walk away, here's what happens:

You'll burn an hour resolving that by hand. The fix is to not let it happen.

The discipline that prevents it

Before you start the agents, write SPEC.md in each worktree, and have each agent read both before planning. Worktree A's plan should know B is happening, and vice versa.

you · worktree A
Write SPEC.md for adding CSV export to the dashboard list endpoint. Important: a parallel branch (refactor/cursor-pagination) is replacing offset-pagination with cursor-pagination repo-wide, including dashboard/list.ts. Read ../myproj-refactor-cursor-pagination/SPEC.md before planning, and design CSV export to work with cursor pagination, not offset. Write the SPEC and stop. No code yet.

The same instruction, mirrored in worktree B's session. Each side knows the other exists. Each side reads the other's plan.

why this works

You moved the coordination problem from code level (merge time) to spec level (plan time). Specs are short; code is long. Resolving a contradiction in two 20-line SPEC.md files takes a minute. Resolving a contradiction in two 200-line diffs takes an afternoon.

The flow, side by side

Worktree A · CSV export

SPEC.md: list.ts gets a ?format=csv query. When set, response is text/csv. Pagination is unchanged in contract — limit/offset stays, but CSV format does not paginate (returns full result). Will gracefully accept either pagination shape during the migration.

Agent reads the cursor-pagination SPEC, notes the migration window, writes its own SPEC to handle both.

Worktree B · cursor pagination

SPEC.md: replace ?page=&limit= with ?cursor=&limit= on all list endpoints. Maintain a one-release deprecation: accept both, return the new shape, emit a deprecation header. Document the migration in the ADR.

Agent reads the CSV SPEC, notes that CSV ignores pagination so it doesn't need cursor changes inside csv.ts.

Now the two agents are working on changes that can compose. Worktree A's CSV format ignores pagination shape. Worktree B's pagination refactor accepts both styles. The merge becomes a straightforward concatenation of edits in list.ts rather than a contradiction.

Watching them work, in parallel

Each session proceeds independently. Three principles to hold:

  1. Do not read both terminals at once. Pick one to babysit; let the other run with autonomy. Your serial attention is the bottleneck, not the agent's. Switch when you hit a natural pause.
  2. Use the reviewer in each worktree. Each agent runs /verify when it thinks it's done. Each reviewer runs the tests in that worktree — no cross-contamination.
  3. Commit incrementally in each. Don't let either branch accumulate four hours of work. Small commits make the merge legible.

The merge

Both branches reach reviewer-clean. Now:

terminal 1 · in worktree A
git push origin feature/csv-export
gh pr create --title "feat(dashboard): CSV export" --body-file PR_BODY.md
# wait for CI
gh pr merge --squash  # or however your team merges

Now worktree B needs to absorb A. Two strategies:

Rebase (preferred for cleanish history)

in worktree B
git fetch origin
git rebase origin/main
# resolve any list.ts conflicts
# if you wrote both SPECs to coexist,
# there usually are none.
git push --force-with-lease

Merge main (lower-risk for big diffs)

in worktree B
git fetch origin
git merge origin/main
# resolve conflicts in the merge commit
git push

If you did the spec-coordination right, there's a tiny conflict at the top of list.ts (imports overlap) and nothing semantic. Resolve in two minutes. Run the reviewer one more time in worktree B. Ship.

What goes wrong, honestly

You will at some point skip the spec-coordination step "because the features feel unrelated." Then you'll get bitten. The standard failures:

FailureDetectionRecovery
Both branches edit the same function with different intent Merge conflict at line level Read both diffs. Write a third version by hand or with the agent. Don't blindly accept either side.
Branch A relies on a helper Branch B deleted Tests pass in each worktree, fail after merge Run /verify on the merge result before pushing. If you only verified per-branch, you didn't verify the integration.
Branch B introduces a new pattern Branch A is using the old pattern of Code review confusion ("why does this use offset?") This is a SPEC.md failure. The agent in A didn't know about B's pattern shift. Update both SPECs and re-plan.
Shared dependency added at different versions in package.json Merge conflict in lockfile Pick one. Re-run pnpm install after merge. Verify.

When NOT to parallelize

  • The two tasks share a hot file. If both will heavily edit the same module, serialize. The merge will cost more than the parallelism saved.
  • One depends on the other. Refactor first, then build on top. Parallel here just produces a guaranteed conflict.
  • You can't actively supervise both. Two unsupervised agents are not "twice the throughput." They're two opportunities for divergence. If you can't watch the second, run it serially after the first.
  • The work is small. Two 30-minute tasks done sequentially with the same agent and the same reviewer are faster than two worktrees set up, two specs written, two PRs opened, and a merge.

Cost & time

SequentialParallel worktrees (with coordination)
Wall clock~3 hours~1h 50m
Tokens (approx)~80K~95K (some duplicated reading)
Cognitive loadModerate — one streamHigher — two streams + a merge
Risk profileLowMedium — semantic conflicts possible

You saved an hour of wall clock. You spent twenty percent more in tokens and added a real merge step. For two features at once, that's usually a win. For four, it stops being one.

the honest ceiling, again

Two worktrees: comfortable. Three: tight. Four: you will mess up a merge. The bottleneck is your attention, not the agents'. Build the discipline of reading both SPECs out loud to yourself before approving either — that habit, more than any tooling, is what makes parallel work safe.

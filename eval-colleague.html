Evaluating a colleague's OpenCode agents

Eleven agent files from messuped/dotfiles. This is not a loose bag of role-named chatbots — it's a disciplined orchestrator-worker pipeline with TDD-first enforcement, dual A/B critics and reviewers on different models, and tight permission scoping. It's the most coherent personal agent setup in these field notes. Here's the full read, file by file.

headline verdict

Genuinely good, and more sophisticated than most published setups. The architecture mirrors what this guide and superpowers both arrive at independently: plan → critique → TDD → implement → test → review, with a single coordinator and fresh-window subagents. The standout idea — A/B pairing the critics and reviewers across two different models (Sonnet 4.6 + GPT-5.4) to cancel single-model bias — is something neither this guide nor superpowers does, and it's worth stealing. The costs are real (token multiplication, orchestrator context pressure, heavy for small tasks) and there are two or three smells worth fixing, but the bones are excellent.

The shape of the system

One primary agent (tech-lead) orchestrates ten subagents. It's a textbook orchestrator-worker with an evaluator-optimizer loop bolted on — and unusually, the evaluator is doubled. Two workflows are encoded: a Jira-ticket-to-code path and a new-feature/refactor path. Both run the same spine:

File-by-file

tech-lead.md — the orchestrator

primary opus 4.6 bash: scoped allow-list task: 11 named subagents

The brain. A primary agent on Opus 4.6 that never writes code itself (delegates all implementation), routes work through the two workflows, and enforces a user-approval gate before any code is written. Strong points: every user-facing question is forced through a structured question tool rather than prose; it carries full context into each delegation (subagents share no memory); and it bakes in a 3-worktree ceiling with a confirmation gate — the exact discipline from step 5, here automated. Its own bash is denied except specific read-only git plus worktree management, and it's explicitly forbidden from committing/pushing unless you ask by name. This is a careful, security-conscious design.

smell

It's a ~500-line system prompt. That's a lot of instruction riding at the top of every session's context, and the orchestrator holds the full brief + every subagent's output across a long multi-step flow — exactly the context-rot pressure the subagents are designed to avoid. The coordinator is the one window that doesn't get to stay fresh. Mitigated by Opus's large window, but on a big feature this is where quality will degrade first.

requirements-clarifier.md — spec extraction

subagent sonnet 4.6 all writes denied temp 0.2

Turns a vague request or Jira ticket into a structured brief (scope, acceptance criteria, edge cases, affected areas). Read-only, no codebase exploration by design — it reasons from the ticket and hands off. Notable discipline: a hard rule that "Open Questions must be genuine blockers," not the agent's own design assumptions — a direct fix for the failure mode where clarifiers invent ceremony. Jira MCP integration is present but commented out. This is the Write strategy done as a dedicated step.

architect-designer.md — high-level design

subagent opus 4.6 bash/edit/task all denied manual-invoke only

The one I flagged first when I could only see this file. Verdict after seeing the whole set: defensible, with caveats. It's a role-named "elite architect with 20+ years' experience" that can't run anything — no verifier by construction. But three things redeem it: it's pinned to a frontier model where the work is genuinely reasoning-bound; it outputs concrete artifacts (Mermaid diagrams, ADRs, directory structures), not vibes; and it's explicitly not in the standard workflow — invoked manually when you want a design pass. So it functions as "plan mode as a callable persona," which is legitimate. The persona padding ("Fortune 500") is theater that costs tokens and adds nothing, but the underlying function is sound.

task-planner.md — decomposition

subagent sonnet 4.6 all writes denied

Breaks a designed feature into sequenced, time-boxed tasks (each <2h, ideally 15–30 min). Read-only; outputs a task list with completion criteria and decision points. Sound, and correctly separated from the architect (design first, then decomposition informed by it). Mild overlap risk with architect-designer, but the tech-lead runs them sequentially with distinct jobs, so it holds.

plan-critic-a.md & plan-critic-b.md — the dual critics ★

subagent ×2 sonnet 4.6 gpt-5.4 all writes denied temp 0.5

The cleverest part of the whole setup. Identical prompts, two different models, run in parallel, neither sees the other's output. The tech-lead then synthesizes — consensus concerns weighted highest, single-source concerns flagged as such. This is a genuine de-biasing technique: single-model critique tends to share the generating model's blind spots, and pairing across vendors cancels some of that. They stress-test the plan before code exists, which is the cheapest possible place to catch a spec error (chapter 03: 79% of failures are spec/coordination). Neither this guide nor superpowers does cross-model A/B. Worth stealing.

why this works

The critics have no verifier — they can't run anything — which would normally be the death knell per this guide's rules. But plan critique is one of the few jobs where there's no ground truth to run against anyway: you're evaluating a plan, not code. Given that, cross-model disagreement is the best available proxy for "what might be wrong here," and using two models is a real improvement over one. This is the correct application of a no-verifier agent.

implementation-specialist.md — the builder

subagent sonnet 4.6 edit: allow bash: allow-list w/ deny rails temp 0.3

Writes code to make the pre-written tests pass. Strong scoping discipline: "change ONLY what you are told," no unrequested refactors, no new deps without approval, explore-before-edit via a read-only explore subagent. The permission block is the right shape — broad bash for build/compile, but git add/commit/reset/push and rm/rmdir explicitly denied, so it can't mutate history or destroy files. Crucially it's told not to run the test suite — that's the test engineer's job — which keeps a clean separation between who writes code and who verifies it. Output is a fixed four-section contract the orchestrator parses. This is a well-built worker.

test-automation-engineer.md — the verifier ★

subagent sonnet 4.6 edit: allow runs the suite temp 0.1

The agent with actual teeth. Two modes: TDD mode writes failing tests from the brief before any implementation exists; Run mode discovers the test command, runs the suite, reports pass/fail with root-cause analysis. This is the genuine external verifier (step 3) — it executes, captures real output, distinguishes test defects from code defects. The TDD-first sequencing (tests written at step 3, before implementation at step 4) is exactly superpowers' best idea, independently arrived at. Same protective deny-rails on git/rm. This is the most valuable file in the set.

review-a.md & review-b.md — the dual reviewers ★

subagent ×2 sonnet 4.6 gpt-5.4 bash/edit/webfetch denied temp 0.1

Same A/B-across-models trick as the critics, applied to code review. They get the branch diff + ticket + test results, and emit severity-tagged findings (CRITICAL/MAJOR/MINOR/NIT) with file:line precision. The tech-lead synthesizes and rolls up severity (takes the higher of the two). Only invoked when the user explicitly asks for review.

the one real weakness

Both reviewers are bash: deny — they review by reading the diff, not by running anything. By this guide's definition (step 3) that's the "reviewer without teeth" pattern. It's partially saved because they receive the test-automation-engineer's real results as input, so ground truth is in their context — they're reading a verifier's output rather than being the verifier. But they can't independently re-run a failing case, probe a hypothesis, or catch a test that passes for the wrong reason. The teeth live in the test engineer; the reviewers are advisory. That's a reasonable division, but know that "review" here means "read," and the only thing actually executing is the test step.

update-agents.md — self-maintenance

subagent haiku 4.5 edit: allow git denied

A nice touch: scans all agent files, runs opencode models, finds outdated model IDs, and (after a confirmation gate) bumps them. Correctly on the cheapest model since it's mechanical. Handles the real maintenance problem of model IDs drifting stale — the kind of cruft chapter 15 warns accumulates. Smart inclusion.

What to steal from it

  1. Cross-model A/B for any no-ground-truth judgment. The paired critics and reviewers on different vendors is the best idea here. For plan critique and code review — where you can't just run a test — two models disagreeing is the strongest signal available. Neither this guide nor superpowers does it. Adopt it.
  2. The structured output contract between agents. Every subagent emits fixed section headings the orchestrator parses. This is what makes the handoffs reliable instead of a game of telephone. It's the Compress strategy as an enforced format.
  3. Per-agent deny-rails on destructive commands. git commit/push/reset and rm denied on every agent that can touch a shell, with the orchestrator holding sole (gated) git authority. Clean blast-radius control.
  4. The self-maintenance agent. A tiny Haiku agent that keeps model IDs current is the kind of thing that prevents slow rot. Cheap to run, easy to add.

What to watch / fix

  1. Give the reviewers teeth, or rename them. Either let review-a/b run targeted commands (re-run a failing test, grep for a pattern) so they can verify their own hypotheses, or be honest that they're "diff readers" and lean on the test-engineer as the real gate. Right now "review" overpromises.
  2. The orchestrator is a context-rot risk. The 500-line prompt plus accumulated subagent outputs across a long feature is the one window that never resets. Consider having subagents write briefs/plans to disk (a docs/ or .plan/ file) so the orchestrator can hold references rather than full text — the Isolate move from chapter 02.
  3. Heavy for small work. The full clarify → design → critique×2 → TDD → implement → test → review×2 march is superb for a real feature and absurd for a one-line fix. The tech-lead does route bug fixes straight to implementation, which helps — but make sure that fast path actually gets used, or small tasks will drown in ceremony (the same caveat as superpowers).
  4. Token cost. Up to seven model invocations per feature, two of them on Opus and several paired across vendors. This is the 15× reality. Justified for high-value tickets; track cost-per-merged-PR (chapter 15) to confirm it's earning out.

bottom line

This is a strong, thoughtfully-engineered setup — closer to a small-team SDLC than a solo toolkit, clearly built around a Jira-driven workflow (note the upstream/develop base, the ticket slugs, the b2b-online repo references). If your work is ticket-shaped and feature-sized, it's excellent and you could adopt it nearly wholesale. If you do a lot of small brownfield fixes, it's heavier than you need — but even then, three ideas transfer directly: cross-model A/B critique, structured inter-agent output contracts, and per-agent destructive-command deny-rails. The one fix I'd push for is giving the reviewers the ability to actually run something, so "review" means more than "read the diff." Run it against chapter 15's kill criterion in ninety days and keep whatever beats your baseline.

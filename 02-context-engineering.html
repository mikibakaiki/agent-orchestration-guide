Context engineering, the operating system

Andrej Karpathy's framing got this right: the LLM is the CPU, the context window is the RAM, and "context engineering" is the OS that decides what's in RAM at any moment. Most of your repetition pain is a RAM-allocation problem.

The shift in name matters

"Prompt engineering" was the right name in 2022, when the unit of work was a single message. "Context engineering" — popularized by Karpathy and Tobi Lütke in mid-2025, formalized in Anthropic's September 2025 post — is the right name now, when the unit of work is a multi-turn session with tools, files, and subagents in play.

Anthropic, Sept 2025

"Prompt engineering refers to methods for writing and organizing LLM instructions for optimal outcomes. Context engineering refers to the set of strategies for curating and maintaining the optimal set of tokens (information) during LLM inference, including all the other information that may land there outside of the prompts."

The CPU / RAM / OS picture

The model can't reason about something that isn't in RAM right now. The skill is putting the right thing in the right place at the right time — and the rest of this page is the four moves you have for doing that.

The four strategies

From Anthropic's post and LangChain's matching summary, every context-engineering technique falls into one of these four:

strategy 01
Write
Persist information outside the context window so the model can fetch it deliberately. AGENTS.md, PLAN.md, docs/notes/, a vector store, a memory service. Anything written down survives a session restart.
strategy 02
Select
Retrieve only what the current task needs, not the whole knowledge base. Grep beats RAG when you know the symbol. Path-specific instructions beat global ones when scope matters.
strategy 03
Compress
Summarize long content before it eats the window. /compact. Subagent reports a 500-token findings file instead of 50K of raw output. Conventional commits over freeform commit prose.
strategy 04
Isolate
Move work into a separate context window. Subagents. Worktrees with separate sessions. The reviewer that doesn't see the implementation history. Each isolated context starts at the top of its own attention curve.

Mapping your stated pain to the strategies

You said: "I find myself repeating a lot of stuff in between prompts and agents — framework, code style, git workflow." That's almost entirely a Write + Select problem.

You keep re-typingMoveWhere it lives
"Use Vite, not Next.js" Write ~/.config/opencode/AGENTS.md (global)
"This repo uses Drizzle, not Prisma" Write AGENTS.md in project root
"Conventional Commits, no force-push" Write ~/.config/opencode/AGENTS.md
"Tests in __tests__/integration/" Write Project AGENTS.md
"For Python files, use uv and ruff" Select .github/instructions/python.instructions.md with applyTo
Project glossary, domain terms Write Project AGENTS.md

You will spend roughly an hour writing these files once. After that, every agent on your machine — Copilot in VS Code, OpenCode in a terminal, Copilot CLI, Claude Code, whatever you adopt next — reads them. The repetition stops because the information is in disk and gets paged into RAM automatically.

why this is the unlock, not the agents

People build elaborate "team-lead agents" trying to enforce code style across other agents. The boring answer is: write the style in a file every agent already reads. The agent doesn't need to remember; the file does.

When Write isn't enough: Select and Isolate

Two cases where you'll outgrow markdown-only:

  • The instruction is too narrow for the whole repo. Solution: path-specific instructions. GitHub Copilot's .github/instructions/*.instructions.md files take a applyTo glob. OpenCode supports per-directory AGENTS.md. The rule lives only where it applies.
  • The task is so big it pollutes the main agent's context. Solution: dispatch to a subagent (Isolate). The subagent reads 50K tokens of code, writes a one-page summary to docs/notes/, and returns. The main agent never paid the 50K.

Compress, used deliberately

Two warnings about compression:

  • /compact is lossy. Don't rely on it for state that must survive — write that to a file first. If your plan only exists in the running session and you compact, the compacted summary may quietly drop the riskiest line.
  • Force the summary to be structured. "Summarize" produces blandness. "List unresolved decisions, list files touched, list assumptions made" produces something you can act on.

The honest cost of context engineering

It is overhead. You will spend cycles maintaining files, tightening AGENTS.md, deciding what to compact. Two things make it worth it:

  1. The files are reusable across tools. One AGENTS.md serves Copilot, OpenCode, Codex, Cursor, Amp, Factory, Jules — the format is now standard. You're not maintaining N tool-specific instruction files.
  2. They version with the code. When the stack changes, the file changes in the same PR. Agents read the current state, not stale instructions in someone's chat history.

When orchestration helps, and when it hurts

Anthropic's multi-agent research system beat single-agent Opus by 90.2% on their internal eval. It also used fifteen times the tokens, and the same post says coding is "not a good fit." Both of those facts matter for what you build.

The headline number, with its caveats intact

From Anthropic's June 2025 post "How we built our multi-agent research system":

Anthropic, in their own words

"Our internal evaluations showed that the multi-agent system with Claude Opus 4 as the lead and Claude Sonnet 4 subagents outperformed a single-agent Claude Opus 4 by 90.2% on our internal research evaluation."

That's the number you'll see in slide decks. Three things from the same post you won't:

  • ~15× the tokens of chat. "Multi-agent systems use about 15× more tokens than chats. For economic viability, the application needs to have a high enough value to pay for this." Research with deep web fan-out can justify it. Most coding tasks cannot.
  • Token usage alone explained ~80% of performance variance. Tool calls and model choice explained the rest. Prompt phrasing was not a primary driver. Translation: a single agent given 4× the token budget would close most of the gap.
  • Coding is explicitly called out as the unfavorable case. "Domains that require all agents to share the same context or involve many dependencies between agents are not a good fit for multi-agent systems today. Coding is one example."

The decomposition test

The one question worth asking before adding an agent:

"Independent strands" has a specific meaning: the strands don't need to read each other's output mid-flight. If subagent B needs subagent A's result before it can act, you don't have parallel work — you have a sequence with extra steps and merge pain.

Two examples, side by side

Splits well · fan out

"Research what each of OpenCode, Cursor, Amp, and Claude Code does about subagents."

Four queries. None depends on another's answer. Four subagents in four windows, each returns a findings markdown. A lead synthesizes. This is the Anthropic research-system shape exactly.

Doesn't split · stay single

"Refactor UserService to use the new auth abstraction, then update its callers, then update the tests."

Each step depends on the last. Splitting means coordinating three agents on a moving target. One agent, plan mode, plus a reviewer at the end is better and cheaper.

The failure taxonomy

The MAST study (UC Berkeley et al., 2025) catalogued failure modes across deployed multi-agent systems. Two findings worth holding onto:

your fault, not the model's — underspecified tasks and broken handoffs between agents.
  1. Most multi-agent failures are upstream of the model. Specification, design, and inter-agent coordination are 79% of the surface area. A better model rarely fixes them; clearer specs and fewer handoffs do.
  2. Adding agents adds coordination overhead nonlinearly. The SILO-BENCH study (2025) ran 1,620 configurations and found a "Communication-Reasoning Gap": agents acquired enough information but failed to integrate it. Past a certain point, coordination tokens eat the gains.

The anthropomorphism trap

You said you want "team-lead agents, architect agents, product-owner agents." This framing feels intuitive because it maps onto org charts you already know. The evidence is unkind to it.

  • MAST found that role-named architectures suffer from "role and task ambiguity" — agents disobey their assigned roles or make unilateral executive decisions. Naming an agent the architect does not make it act like one. It adds a constraint the model occasionally violates.
  • Shrivu Shankar (ex-Anthropic, extensive Claude Code experience) argues that custom subagents "gatekeep context" — the main agent loses the ability to reason holistically about a change — and "force human workflows" onto a system that should self-orchestrate. His preferred pattern is one main agent that dispatches generic clones via the Task tool.
  • Even Geoffrey Huntley, who built one of the most ambitious public agentic workflows: "Ralph is monolithic. At this stage, [multi-agent coordination] is not needed."
a better mental model

Build agents by information flow and verifiability, not by job title. For each candidate agent, name (a) the specific information-flow bottleneck it removes, and (b) the verifier it adds. "Because real teams have a PO" is not a reason. "It runs pnpm test in its own window and returns structured failures" is.

Where multi-agent actually pays off, for one developer

The honest list, ranked:

  1. Reviewer / evaluator-optimizer. One agent writes code; another runs tests and reports. Fresh context for the reviewer means it isn't rationalizing from the same starting point. This is the highest-ROI multi-agent pattern for coding work.
  2. Explorer / read-only fan-out. Mapping an unfamiliar codebase. Three small subagents each take a subsystem and write a findings file. Cheap, easy to verify (the files are the artifact).
  3. Plan-then-implement, when the plan is risky. Not really "two agents" — two phases. But you get the same context-isolation benefit if the plan is reviewed before the implementer reads it.
  4. Research fan-out. If your knowledge work involves multi-source synthesis (it does, per your brief), this is where you'll feel the 90.2% kind of effect, not in coding.

Things not on that list: a "team-lead agent" that manages other agents in a long-running role; a "PO agent" that interrogates you about acceptance criteria; an "architect agent" that pre-thinks about every change. These are roleplay. They don't have a verifier; they don't carry new information.

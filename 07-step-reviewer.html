Build · step 03 · ~20 min

Build a reviewer agent with teeth

The single highest-ROI subagent. Runs in its own context window. Executes — does not read. Returns structured failures. If you only build one agent, build this one.

What "with teeth" means

A reviewer agent that reads code and says "looks good" is theater. A reviewer agent that runs the test suite, the linter, and the typechecker and reports their structured output is a real Reflexion-style verifier. The difference is whether the agent can execute commands.

The OpenCode config

Save this as ~/.config/opencode/agents/reviewer.md. The filename is the agent name; you'll invoke it as @reviewer.

~/.config/opencode/agents/reviewer.md
---
description: Runs tests, lint, and typecheck. Reports failures with file:line precision.
mode: subagent
model: anthropic/claude-haiku-4-5
temperature: 0.1
tools:
  bash: true
  read: true
  write: false
  edit: false
permission:
  bash:
    "pnpm test*": allow
    "pnpm lint*": allow
    "pnpm typecheck*": allow
    "npm test*": allow
    "uv run pytest*": allow
    "ruff check*": allow
    "mypy*": allow
    "*": deny
---
You are a code reviewer. You do not modify code. Your job:

1. Detect the project's verification commands from AGENTS.md or package.json.
   Common ones: `pnpm test:ci`, `pnpm lint`, `pnpm typecheck`, `uv run pytest`,
   `ruff check`, `mypy --strict`.

2. Run them in this order, capturing full output:
   a. Typecheck (cheapest, catches most)
   b. Lint
   c. Tests

3. Emit a structured report in this exact format:

   ```
   ## Reviewer report

   - [x|✗] typecheck — <one-line summary>
   - [x|✗] lint — <one-line summary>
   - [x|✗] tests — <one-line summary>

   ### Failures

   1. <file>:<line> — <the exact error>
      hypothesis: <one-line guess at the root cause>

   2. ...
   ```

4. Do NOT modify code. Do NOT suggest patches.
   Just describe what is broken, where, and your single best hypothesis.

5. If everything passes, say so in one sentence and exit.

6. If a command isn't available, say so explicitly — do not pretend it passed.

Reading the config, line by line

mode: subagent

Tells OpenCode this runs in its own context window when invoked from another agent. Fresh attention budget every call.

model: haiku-4-5

Cheap and fast — the work is mostly running commands and reading their output. Reasoning quality is rarely the bottleneck here.

temperature: 0.1

Near-deterministic. We want consistent, factual reporting — not "creative" interpretations of stack traces.

tools: bash, read

Bash to execute commands. Read to look at source when explaining errors. Write and edit are explicitly off. The reviewer can't fix; only report.

permission allow-list, then deny-all

The most important block. Bash only for the specific verification commands. Everything else is denied. This is not paranoia — this is what lets you say tools: bash: true without it being equivalent to handing the agent a shell. If the agent tries rm -rf or curl-to-anywhere, it's blocked.

Using it

From the main OpenCode session, after the implementer makes a change:

main session prompt
I've finished the auth refactor. Hand it to @reviewer.

The main agent invokes the reviewer subagent in a fresh window. The reviewer runs the commands, returns its structured report, and the main agent reads it. Crucially, the main agent's window doesn't pay for the test output — only the summary.

What the report looks like in practice

@reviewerhaiku-4-5 · 2.4s · 1,200 tokens
## Reviewer report - ✓ typecheck — clean - ✗ lint — 2 errors in src/server/routes/auth/ - ✗ tests — 1 failing in __tests__/integration/auth.test.ts ### Failures 1. src/server/routes/auth/handler.ts:42 — '_userId' is declared but never read (TS6133) hypothesis: leftover from earlier draft; remove the parameter or prefix with underscore consistently. 2. __tests__/integration/auth.test.ts:88 — expected status 401, received 500 hypothesis: the new error handler is throwing before the auth middleware fires; likely missing await on `validateToken()` in handler.ts:31.

The hypothesis line is the part the model is uniquely good at — it can read stack traces and connect them to recent diffs. Don't trust the hypothesis blindly; do trust the file:line.

The Claude Code equivalent

Claude Code uses .claude/agents/ for subagents with a similar YAML frontmatter. The shape:

.claude/agents/reviewer.md
---
name: reviewer
description: Runs project verification (tests, lint, typecheck) and reports failures.
tools: Bash, Read
---

[same body as above]

Invocation is similar — from the main session, Use the reviewer agent to check this.

Common pitfalls

  • Leaving edit: true on. The reviewer will "helpfully" fix things, defeating the point. The implementer fixes; the reviewer reports.
  • Permission allow-list too broad. If you allow "*": allow for bash, you've made the reviewer a shell. Whitelist specific commands.
  • No mention of AGENTS.md. The reviewer needs to know where to look for the verification commands. Either reference AGENTS.md in the prompt or pin commands explicitly (as in the template above).
  • Tone too friendly. "Looks great!" reports aren't useful. The prompt above forces a structured format because free-form reviews from a chatty model are not actionable.
  • Forgetting to make tests deterministic. If your tests are flaky, the reviewer will flap. Fix the flakes first.

One harder variant: an "independent" reviewer

Anthropic's Claude Code best practices note: "Ensuring independent verification by sub-agents helps avoid overfitting to tests." The pattern: have a second reviewer that does not run the test suite, but instead reads the diff and the spec, and answers "does this diff actually implement what the spec asked for?"

It's a higher-variance check (no ground-truth signal), so use it only on diffs you want to be careful about — and don't trust the result as much as the test-runner's. But it catches a class of failure the test-runner can't: "the tests pass because the agent wrote tests that match its broken implementation."

What this earned you

  • A fresh context window for verification, every time. The reviewer doesn't accumulate session noise.
  • A structured report you can read in five seconds.
  • A real external verifier — the test suite — driving the loop.
  • An explicit "implementer fixes, reviewer reports" division that prevents the model from cheerfully editing tests to pass.

You will use this agent dozens of times a week. It's the workhorse.

Field note · synthesis

Build your own: the best of both

You've seen superpowers (skills-first, mandatory methodology) and your colleague's setup (agents-first, cross-model A/B). Here's how to take the strongest idea from each — plus this guide's lean foundation — into one personal OpenCode setup. Full, copy-able code. Start minimal; grow only what earns its place.

The design, in one breath

what we're building

A three-layer stack: (1) a lean AGENTS.md as the always-on foundation; (2) two skills stolen from superpowers — TDD-first and spec-first — as on-demand procedures; (3) three agents — a reviewer with real teeth, an explorer, and a cross-model second-opinion reviewer borrowed from your colleague. Plus the permission rails and structured-output discipline that made the colleague's setup safe. No orchestrator, no role-play personas, no ceremony you won't use.

Layer 1 — the foundation: AGENTS.md

The always-on knowledge. Note the two stolen rules baked right in near the top: tests first and spec to disk for non-trivial work. These make the skills' behavior the default even before a skill loads.

~/.config/opencode/AGENTS.md
# Personal engineering standards

## Two hard rules (stolen, on purpose)
1. **Tests first for non-trivial work.** Write a failing test, watch it
   fail for the right reason, then implement. If you wrote code before
   its test, delete the code and start again. (From superpowers.)
2. **Spec to disk before multi-file work.** Anything touching >3 files
   or introducing a new pattern gets a short SPEC.md written and shown
   to me BEFORE implementation. No code until I approve it.

## Languages & defaults
- TypeScript strict, no `any`. Python 3.12 via uv, ruff, mypy --strict.
- Prefer Vite + React; Fastify or FastAPI for servers. Not Next/Express.
- Postgres + Drizzle or SQLAlchemy 2.x. Use the repo's existing migrator.

## Code style
- Functions over classes unless state demands it. Early returns.
- Comments explain *why*, not *what*. No emoji in code or commits.
- Match the surrounding file's style over my defaults when they conflict.

## Git workflow
- One logical change per commit. Conventional Commits.
- Never force-push shared branches. Never commit secrets or .env.
- You do NOT run git commit/push/reset unless I ask by name.

## When unsure
- Read the codebase first; match existing patterns.
- Ask before adding a dependency or refactoring >5 files.
- Surface genuine blockers as questions; don't invent answers.

## Structured output (stolen from a colleague's setup)
When you finish an implementation, end with these exact sections so
tools and I can parse them:
### What was done
### Files changed
### How to verify
### Remaining concerns

Layer 2 — skill: TDD

The superpowers idea, distilled to one on-demand file. Loaded when a task is implementation-shaped; it's a procedure the working agent follows in-place, not a separate window.

~/.config/opencode/skills/tdd.md
# Skill: test-driven development

Apply this when implementing any non-trivial behavior change.

## The loop (do not skip steps)
1. **RED** — write the smallest failing test that expresses the next
   piece of desired behavior. Run it. Confirm it fails, and that it
   fails for the *expected* reason (not a typo or import error).
2. **GREEN** — write the minimum code to make that test pass. Nothing
   more. Resist implementing the next feature.
3. **REFACTOR** — clean up while the test stays green. Then commit.
4. Repeat for the next behavior.

## Rules
- If you find yourself writing production code with no failing test
  driving it, STOP. Delete it. Write the test first.
- One behavior per test. Name the test after the behavior, not the method.
- Cover the error/edge cases the spec lists, not just the happy path.
- Tests are code: same quality bar, no flakiness, no magic numbers
  without a comment.
- Do NOT weaken a test to make it pass. If the test is wrong, fix the
  test deliberately and say so.

## When NOT to use this
Trivial one-line fixes, pure renames, comment/doc changes. Use judgment;
the goal is confidence, not ceremony.

Layer 2 — skill: writing specs

~/.config/opencode/skills/writing-specs.md
# Skill: writing a SPEC before implementation

Apply this before any change touching >3 files or adding a new pattern.

## Produce a SPEC.md in the repo with these sections:

### Goal
One paragraph: what changes for the user, and why.

### Non-goals
What this explicitly does NOT do. Bound the scope.

### Design
Affected files/modules. Chosen approach. Patterns to follow (point at an
existing example in THIS codebase). Data model changes if any.

### Test plan
The acceptance criteria as a checklist. Each becomes at least one test
in the TDD step.

### Risks & open questions
Anything you had to assume. Anything only I can decide. Keep open
questions to genuine blockers — not your own design preferences.

## Then STOP.
Show me the SPEC and wait for approval. Do not write code until I say go.
The cheapest place to fix a mistake is in the spec, before any code exists.

Layer 3 — agent: reviewer with teeth

This guide's step 3 reviewer, hardened with the colleague's deny-rails. Unlike the colleague's read-only reviewers, this one runs commands — that's the whole point. Fresh window, executes the verifiers, reports structured failures, fixes nothing.

~/.config/opencode/agents/reviewer.md
---
description: Runs tests, lint, and typecheck in a fresh window. Reports
  failures with file:line precision. Does not modify code.
mode: subagent
model: anthropic/claude-sonnet-4-5
temperature: 0.1
permission:
  edit: deny
  webfetch: deny
  bash:
    "pnpm test*": allow
    "pnpm lint*": allow
    "pnpm typecheck*": allow
    "npm test*": allow
    "uv run pytest*": allow
    "ruff check*": allow
    "mypy*": allow
    "tsc --noEmit*": allow
    "git diff*": allow
    "git log*": allow
    "*": deny
---
You are a code reviewer with teeth. You RUN the verifiers — you do not
just read the diff.

## Process
1. Detect the project's verify commands from AGENTS.md or package.json.
2. Run them, capturing full output, in this order:
   typecheck → lint → tests.
3. Emit this exact structure:

## Reviewer report
- [x|✗] typecheck — 
- [x|✗] lint — 
- [x|✗] tests — 

### Failures
1. : — 
   hypothesis: 

## Rules
- Do NOT edit code. Do NOT suggest patches. Report only.
- If a command is unavailable, say so — never pretend it passed.
- If everything passes, say "verified" in one line and stop.

Layer 3 — agent: cross-model second opinion

The colleague's best idea, kept lean. A second reviewer on a different model, for the diffs you want extra eyes on. It reads rather than runs (so it's advisory) — you pair it with the teeth-reviewer above when a change is high-stakes, and the disagreements between them are the signal.

~/.config/opencode/agents/reviewer-b.md
---
description: Independent second-opinion reviewer on a different model.
  Reads the diff and the teeth-reviewer's results; flags what a
  single model might miss. Use for high-stakes changes.
mode: subagent
model: openai/gpt-5.4          # deliberately a different vendor
temperature: 0.1
permission:
  edit: deny
  bash: deny
  webfetch: deny
---
You are an independent reviewer. You have NOT seen any other reviewer's
reasoning — only the diff and the test results you're given. Your value
is catching what a single model's blind spots would miss.

You will receive:
- BRANCH DIFF
- TEST RESULTS (from the teeth-reviewer, if available)
- SPEC (if one exists)

Review for: correctness, edge cases, security, and — critically —
**whether the tests actually prove the behavior, or just pass.** A test
suite can be green and still wrong. That's the failure the runner can't
catch and you can.

## Output
### Summary
One paragraph.
### Issues
**[CRITICAL|MAJOR|MINOR|NIT] title**
- File: `path:line`
- Detail: what's wrong and why it matters
- Suggestion: specific fix
### Positives
Specific, not generic.
### Open questions
Genuine blockers only.
why two reviewers, not one

The teeth-reviewer gives you ground truth (does it build, do tests pass). The second-opinion reviewer, on a different vendor's model, catches the thing ground truth can't: tests that pass for the wrong reason, missed edge cases, security smells. You don't run both on every change — only when the stakes justify the tokens. For routine work, the teeth-reviewer alone is enough.

Layer 3 — agent: explorer

~/.config/opencode/agents/explorer.md
---
description: Maps unfamiliar code in a fresh window. Read-only. Writes a
  findings file and returns a 5-bullet summary.
mode: subagent
model: anthropic/claude-haiku-4-5    # bandwidth-bound → cheap model
temperature: 0.2
permission:
  edit: deny
  write: allow          # only to write the findings file
  webfetch: deny
  bash: deny
---
You investigate code without changing it. Given a topic:

1. Find the entry points for that area.
2. Trace 2-3 representative call paths to the data layer.
3. Note key types, invariants, and TODO/FIXME hints.
4. Write findings to `docs/notes/explore--.md`:
   ## Scope / ## Entry points / ## Key files / ## Data model /
   ## Invariants / ## Gotchas / ## Open questions
5. Return ONLY: the file path + a 5-bullet summary.

Never edit source. Never run commands. If write fails, return the
markdown inline with a note.

The commands that tie it together

~/.config/opencode/commands/verify.md
---
description: Run the reviewer with teeth on the current diff.
---
Hand the current diff to @reviewer. Summarize: what passed, what failed
(with file:line), and a one-line fix plan per failure. If all pass, say
"verified" and stop. For a high-stakes change, also invoke @reviewer-b
in parallel and tell me where the two reviewers DISAGREE — that's the
signal worth my attention.
~/.config/opencode/commands/spec.md
---
description: Write a SPEC to disk before implementation.
---
Apply the writing-specs skill. Produce SPEC.md for "$ARGUMENTS",
show it to me, and STOP. Do not write any code until I approve.
~/.config/opencode/commands/explore.md
---
description: Map an unfamiliar area of the codebase.
---
Hand "$ARGUMENTS" to @explorer. When it returns, give me the 5-bullet
summary and the path to the findings file.

How a real task flows through it

Medium feature, the way this stack handles it — note how little ceremony there is compared to the full orchestrator:

/spec add rate limiting

The writing-specs skill fires. A SPEC.md lands on disk; the agent stops and shows you. You tweak one line, approve. Cost so far: one window, cheap.

"implement it, TDD"

The tdd skill fires. The same agent writes failing tests, watches them fail, implements, commits incrementally. AGENTS.md already made this the default; the skill reinforces the discipline. Still one window.

/verify

@reviewer spins up in a fresh window, runs typecheck/lint/tests, returns a structured report. You see real pass/fail, not opinions. If it's a high-stakes change, @reviewer-b joins on a different model and you get the disagreements.

you commit

You — not an agent — run the commit, because AGENTS.md forbids agent git operations unless asked. The structured "What was done / Files changed" output makes your commit message and PR body trivial to write.

Three agents touched, two of them only when stakes were high, and most of the work happened in one well-informed window. That's the best-of-both: superpowers' discipline (TDD, spec-first) without its mandatory ceremony; the colleague's safety and cross-model rigor (deny-rails, A/B review) without the heavyweight orchestrator.

The growth path

Don't build all of this on day one. The cost-ordered progression from the previous note applies:

WeekAddWhy now
1AGENTS.md + opencode.json onlyThis alone kills the repetition problem. Live on it.
2The reviewer with teeth + /verifyHighest-ROI agent. The one piece of multi-agent with the most evidence.
3The tdd + writing-specs skillsOnce you trust the loop, add the discipline layer.
4The explorer + /exploreWhen you hit an unfamiliar codebase and feel the main window bloat.
When stakes risereviewer-b (cross-model)Only when a single model's blind spots actually cost you. Not before.
and the kill criterion still applies

Ninety days in, run chapter 15's test: does the full stack beat AGENTS.md-plus-teeth-reviewer alone by a clear margin on your real tasks? If reviewer-b never catches anything the teeth-reviewer missed, drop it. If the skills are ceremony you route around, fold their one good rule into AGENTS.md and delete them. Keep only what earns its tokens.

Claude Code variant

If you're on Claude Code instead of OpenCode, the same design maps over: AGENTS.md → CLAUDE.md; skills/ → Claude Code skills; agents/ → .claude/agents/ with the same YAML frontmatter (name, description, tools: Bash, Read); commands/ → .claude/commands/. The cross-model reviewer-b is the only piece that needs adapting — Claude Code can't natively call a non-Anthropic model, so you'd run that second opinion as a separate manual step or via an API-backed skill. Everything else is a rename.

the through-line

Skills make one mind better-informed; agents bring in fresh minds. The best personal setup uses the cheapest primitive that solves each problem — always-on text first, on-demand skills next, fresh-window agents last — and steals the two or three genuinely great ideas from elaborate frameworks without inheriting their weight. You now have all the code to do exactly that.

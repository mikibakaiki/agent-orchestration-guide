Git worktrees for parallel work

Two agents on the same checkout will trash each other's files. Worktrees give each one its own working directory backed by the shared .git database. Three commands and a shell helper later, you have safe parallel work.

The problem worktrees solve

You're running two agents in two terminals. Both call git checkout. Both edit src/server/auth.ts simultaneously because neither knows about the other. Your filesystem becomes a corrupted superposition.

A worktree is a second working directory linked to the same git repo. Each worktree has its own checkout, its own branch, its own modified files — but shares the underlying .git object database. No duplication of history. No merge collisions during work; you only merge at the end, deliberately.

The three commands

bash
# Create a new worktree on a new branch
git worktree add ../proj-feature-auth -b feature/auth

# Create a worktree from an existing branch
git worktree add ../proj-bugfix-403 bugfix/403

# See what worktrees exist
git worktree list

# Remove a worktree (cleans up the directory)
git worktree remove ../proj-feature-auth

# Prune stale entries after deleting a worktree directory manually
git worktree prune

That is the whole feature. Now wrap it in three shell functions so you don't have to think about paths.

The shell helpers

Drop these in your ~/.bashrc or ~/.zshrc:

bash · ~/.bashrc or ~/.zshrc
# wt <branch> — create or enter a worktree for a branch
wt() {
  local repo branch dir
  repo="$(basename "$(git rev-parse --show-toplevel 2>/dev/null)")"
  if [ -z "$repo" ]; then echo "not in a git repo"; return 1; fi
  branch="$1"
  if [ -z "$branch" ]; then
    git worktree list
    return 0
  fi
  dir="../${repo}-${branch//\//-}"
  if [ ! -d "$dir" ]; then
    git worktree add "$dir" -b "$branch" 2>/dev/null \
      || git worktree add "$dir" "$branch"
  fi
  cd "$dir"
}

# wtrm <branch> — remove a worktree cleanly
wtrm() {
  local branch="$1"
  local repo
  repo="$(basename "$(git rev-parse --show-toplevel)")"
  git worktree remove "../${repo}-${branch//\//-}" \
    && git branch -D "$branch" 2>/dev/null
}

# wtclean — prune merged/stale worktrees
wtclean() {
  git worktree prune
  git worktree list
}

The branch-to-directory mapping (feature/auth → ../proj-feature-auth) is deliberately predictable so the agent can read its own working directory and infer its branch.

The daily flow

bash · two terminals
# terminal 1
cd ~/code/myproj
wt feature/payments        # creates ../myproj-feature-payments, cd's in
opencode                   # or claude, or copilot, or codex
# ...work, commit, push, open PR...
cd - && wtrm feature/payments

# terminal 2 (in parallel with terminal 1)
cd ~/code/myproj
wt bugfix/timeout
opencode
# ...work...

What the agent should know about worktrees

Add a line to your project AGENTS.md:

AGENTS.md · add this
## Worktrees
This repo uses git worktrees for parallel agent work. Each worktree
sits in a sibling directory named `<repo>-<branch>`. You're working
in whichever worktree the session started in — derive your branch
from `git branch --show-current`, not from assumptions.

Do not run `git checkout` to switch branches mid-session. If you need
a different branch, exit; the user will run `wt` to switch worktrees.

This prevents the agent from helpfully switching branches inside an active session, which would crash everything you're doing in that worktree.

The honest ceiling

Worktrees scale to filesystem parallelism trivially. They do not scale to cognitive parallelism.

two is plenty; three is the ceiling

The bottleneck isn't running agents in parallel — it's reviewing their output in parallel. By worktree three, you're context-switching between three diffs, three sets of test failures, three near-finished branches. The wall clock you save by parallelizing is eaten by the cognitive cost of holding three things in your head.

Anthropic's docs and Boris Cherny's notes recommend 2–4 worktrees as a soft cap. For one developer, my honest recommendation is two, occasionally three, never four.

What worktrees don't solve

  • Semantic merge conflicts. Two agents writing related features will produce code that doesn't compose, and you'll find out at merge. Worktrees prevent filesystem collisions, not API design collisions. Tell the agents up front what each is doing, in writing.
  • Shared resources. Database, test ports, fixtures. If both worktrees run pnpm dev on port 5173, second one fails. Use distinct ports per worktree, or set PORT per-session.
  • Uncommitted state on main. If you have uncommitted changes on the main worktree, git worktree add creates a new clean worktree — your uncommitted work stays on main, but the new worktree doesn't see it. Commit or stash before creating a worktree if you want the state to travel.
  • node_modules / venvs. Each worktree needs its own. pnpm install per worktree. Some people ln -s ../proj/node_modules ./node_modules to share; risky if versions differ across branches. Cleaner: just install per worktree and accept the disk cost.

Cleanup discipline

The biggest worktree footgun is leaving them around. They accumulate. git worktree list becomes a graveyard of merged branches.

Habit: at the end of every day, wtclean. Look at the list. Remove anything whose branch has been merged or abandoned. Five seconds. Saves you the moment in a month where you realize you have 14 stale worktrees and can't remember which has the half-finished refactor.

on the value of worktrees specifically for agents

Even if you only ever run one agent at a time, worktrees are useful: they let you walk away from a session mid-task without losing main. Start a Ralph loop in a worktree, leave it running, work on something else in your main checkout. The worktree provides clean isolation; the agent has its own playground.

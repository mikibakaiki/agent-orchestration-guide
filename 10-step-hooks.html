Build · step 06 · ~20 min

Hooks, slash commands, auto-docs

The final step. Tie the docs-writer to git events so documentation maintains itself. Make slash commands for the verification and ADR flows you'd otherwise type twice a day. Treat your agent setup like code.

The principle

Docs should be a side effect of work, not a separate item on your list. The moment "I should update the README" becomes a thing you have to remember, it stops happening. Hooks fix that by tying the doc update to an event that's already happening — a commit — so it can't be skipped.

The post-commit hook

.git/hooks/post-commit · chmod +x
#!/usr/bin/env bash
# Auto-doc trigger. Runs after every commit on your local machine.
# Skip if the commit was made by an agent to avoid recursion.

LAST_AUTHOR=$(git log -1 --format='%an')
case "$LAST_AUTHOR" in
  *Claude*|*Copilot*|*GPT*|*bot*) exit 0 ;;
esac

# Skip if the diff doesn't look like user-visible change.
CHANGED=$(git diff HEAD~1 HEAD --name-only 2>/dev/null)
if ! echo "$CHANGED" | grep -qE '(^src/|^api/|^db/|package\.json|pyproject\.toml|migrations/)'; then
  exit 0
fi

# Fire the docs-writer asynchronously so it doesn't slow your commit.
nohup opencode run --agent docs-writer \
  --prompt "Review HEAD. Update README and CHANGELOG if the diff justifies it. Touched files: $CHANGED" \
  > /tmp/docs-writer.log 2>&1 &

The three guards matter:

  • Skip agent commits. Otherwise the docs-writer's own commits trigger another docs-writer run → infinite loop.
  • Skip non-user-visible changes. A pure test refactor doesn't need README touch.
  • Run async. Your commit returns instantly; the docs run in the background. If it fails, the log file tells you why.
portability

.git/hooks/ is local-only; it doesn't survive a clone. For team-shared hooks, use husky, lefthook, or pre-commit — they store hooks in the repo and install them on npm install or equivalent.

Slash commands

If you find yourself typing the same prompt twice a day, make it a slash command. Both Claude Code and OpenCode look for command files in .claude/commands/ and .opencode/commands/ respectively.

/verify · run the reviewer

.opencode/commands/verify.md
---
description: Run reviewer agent for full project verification.
---
Hand the current diff to @reviewer. Wait for the report and summarize:
- which checks passed
- which failed, with file:line
- a one-line plan for fixing each failure

If everything passes, just say "verified" and stop.

Now /verify in any session runs the whole loop. No typing.

/adr · start a new architecture decision

.opencode/commands/adr.md
---
description: Draft an ADR from a short prompt.
---
You're drafting an Architecture Decision Record. Ask me ONE clarifying
question at a time, briefly, until you have:

1. The decision being made (one imperative sentence)
2. The context: what prompted this
3. Two or three alternatives that were considered
4. The expected consequences (positive, negative, to-monitor)

When you have all four, draft `docs/adr/NNNN-<slug>.md` using
the MADR-lite template (status: Proposed, today's date).

Do not invent context I didn't give. If I don't answer one of the
four pieces, leave it as `<!-- TODO -->`.

/explore · dispatch the explorer

.opencode/commands/explore.md
---
description: Map an unfamiliar code area. One-arg: the topic.
---
Hand "$ARG" to @explorer. When it returns, summarize its findings
in 3 bullets and tell me the path of the file it wrote.

/release · narrative changelog

.opencode/commands/release.md
---
description: Convert the [Unreleased] CHANGELOG section into release notes.
---
Read CHANGELOG.md. Take the [Unreleased] section. Produce a short
release-notes paragraph (max 4 sentences) suitable for a release PR.
Group by impact: what users will notice, what fixed, what's internal.
Do not invent items beyond what's in the CHANGELOG.

Five commands. None of them require new infrastructure. Each replaces a paragraph of typing.

Changelog: tools do the deterministic part, the agent does the narrative

Don't ask the agent to "generate the changelog from git log." It will hallucinate. Use a deterministic tool first:

bash · install once
# git-cliff: derive raw changelog from conventional commits, no LLM
brew install git-cliff
# or:
cargo install git-cliff

# init config in your repo
git cliff --init

# generate from commits
git cliff -o CHANGELOG.md

Now the structure of the changelog comes from your commit messages (which are already structured if you're following Conventional Commits). The agent's job — via /release — is only to write the narrative summary on top, not to invent the bullet list.

README staleness check

One specific docs-writer rule worth adding to its prompt: maintain a "Last reviewed against <sha> on <date>" line at the top of the README. If the README hasn't been reviewed in 30+ commits, the docs-writer's hook flags it — either by adding a STALE marker or by opening an issue.

add to ~/.config/opencode/agents/docs-writer.md
If README.md has a line matching `Last reviewed against` and the
referenced sha is more than 30 commits behind HEAD, replace that line
with `Last reviewed against <current-sha> on <today>` and update
the README with anything from the intervening diff that matters.

If README.md has no such line, add one at the top.

Cost monitoring · because hooks fire

Every commit now triggers an agent. That's good for docs and dangerous for your bill if you commit twenty times a day.

  • Use Haiku for the docs-writer trigger, not Sonnet, unless you're explicitly working on something architectural. The hook prompt can default to a small model and only escalate when the diff is large.
  • Tail /tmp/docs-writer.log after a busy day. Watch for the agent making spurious or redundant updates — that's a tuning signal.
  • Cap the hook: if the diff is over N files, the hook skips and asks you to invoke /docs manually. Large refactors deserve human-driven doc updates anyway.

Where this leaves you

If you've done steps 1 through 6:

  • One AGENTS.md per project, plus a global personal file. Three tools read them. Repetition gone.
  • Three subagents: reviewer (verifies), explorer (maps), docs-writer (maintains).
  • Worktrees and shell helpers for parallel work, capped at two-to-three.
  • A post-commit hook that keeps documentation alive.
  • Five slash commands for the workflows you'd otherwise re-type.

That is the full stack. The next four pages walk through it in action, from a 10-minute bug fix up to an overnight Ralph loop in a sandbox.

remember this in two months

When you read your AGENTS.md in two months and half of it feels stale or wrong, that's the system working correctly. The file is meant to drift with the project. Cut what no longer applies; add what you keep correcting. The boring file is a living file.

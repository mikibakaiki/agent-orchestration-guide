Example B · medium

Ship a feature with TDD, review, & docs

Three agents. Six commits. One merged PR. A real feature — rate-limiting on the public API — done end-to-end with plan mode, the reviewer, and the docs-writer triggered by the commit hook. Watch how the agents stay in their lanes.

The scenario

Your API is getting abused. You need to ship per-token rate limiting: 60 requests/minute for free-tier tokens, 1000/minute for paid. Returns 429 with Retry-After header. Configurable per-token. Must be observable (Prometheus counter). Plus tests, an ADR, and updated docs.

stack assumed

Steps 1–6 are all in place. AGENTS.md, reviewer, explorer, docs-writer, post-commit hook, slash commands. Single worktree (this isn't parallel — that's Example C).

Phase 1 — Plan mode, written down

Before writing code, write the plan to disk. Shift+Tab puts Claude Code into Plan Mode; OpenCode has a /plan equivalent. The plan becomes SPEC.md in the worktree.

youplan mode
Plan: per-token rate limiting. 60/min free, 1000/min paid. 429 with Retry-After. Configurable per-token via the existing tokens table. Prometheus counter (rate_limit_hits_total) labeled by tier and route. ADR for the storage choice (Redis vs in-memory). Write SPEC.md with the plan. Don't touch code yet.
agentsonnet 4.5 · plan mode · 32s

Skeleton: # Per-token rate limiting ## Goal ## Non-goals ## Design - storage: Redis (rejected in-memory because we run 3 instances) - sliding window: redis-rate-limiter package, no DIY - middleware sits AFTER auth, BEFORE handler ## API surface - request → header X-RateLimit-Limit, X-RateLimit-Remaining - over limit → 429 + Retry-After ## Test plan - integration test per tier - test for header presence on success - test for retry-after math ## Observability - prometheus counter ## Open questions - what tier do unauthenticated requests get? (writing as tier=anonymous, 30/min — flag this for review) Should I proceed to implementation, or want to edit SPEC.md first?

The "open questions" line is the move. The agent flagged a thing it had to guess, instead of confidently guessing in silence. You get the choice now, cheap, before any tests are written against the wrong answer.

you
Unauthenticated stays at the existing nginx-level limit; don't add a tier for it. Drop "tier=anonymous" from the plan. Proceed.

Phase 5 — The hook fires

That commit triggers your post-commit hook. The docs-writer kicks off async; you don't see it block. Half a minute later, in /tmp/docs-writer.log:

@docs-writerhaiku-4-5 · async, via post-commit hook
Reviewed HEAD~5..HEAD. Touched files include src/server/middleware/, package.json, docs/adr/. Updates made: - README.md: added "Rate limiting" subsection under "API" — 3 lines describing limits per tier and the Retry-After header. - CHANGELOG.md: appended to [Unreleased]: ### Added - Per-token rate limiting: 60 req/min (free), 1000 req/min (paid). Returns 429 with Retry-After header. ### Changed - (none, internal middleware addition) Did NOT modify any source files. Did NOT add claims I couldn't verify from the diff. ADR 0007 already exists and links to the rate-limit middleware, so no further docs work needed. commit: docs: README and CHANGELOG for per-token rate limiting (commit authored as "Claude (docs-writer)" — the post-commit hook will not re-trigger.)

This is the loop closing without you. You committed the ADR; the hook noticed user-visible files changed; the docs-writer scoped its update; it committed with an agent author so the hook doesn't recurse.

Six commits. Each does one thing. The history reads like a story. None of this required you to remember "I should write an ADR" or "I should update the README" — both happened because the system fired.

Who did what

AgentContext windowWhat it touched
Main (sonnet 4.5) Single long session, ~38K tokens by end SPEC.md, the test file, the middleware, app.ts, metrics.ts, the lint fix, the PR. Drove the work.
@reviewer (haiku 4.5) Two fresh windows, ~3K each Ran tests + lint + typecheck twice. Caught the lint warning. Flagged the slow test.
@docs-writer (haiku 4.5) One fresh window, ~5K Read the diff range, scoped updates to README + CHANGELOG. Committed those.

Total cost (approx, Anthropic pricing late-2025 rates)

  • Main agent: ~38K tokens · Sonnet · roughly $0.12
  • Reviewer (2 calls): ~6K tokens · Haiku · roughly $0.01
  • Docs-writer: ~5K tokens · Haiku · roughly $0.01
  • Total: ~$0.14 for a complete, documented, reviewed feature.

Wall clock: ~30 minutes including your interactions. The same feature done by hand — design, tests, implementation, ADR, README, CHANGELOG — is at least an hour, more likely two.

what to take from this

The pattern that did the heavy lifting was plan → SPEC.md → tests → implementation → reviewer → ADR → docs-writer trigger. None of it is exotic. It's the boring sequence happening reliably because the files and hooks are doing the remembering for you. The next example shows what changes when two of these need to run at the same time.

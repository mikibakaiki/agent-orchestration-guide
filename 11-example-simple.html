Example A · simple

A 10-minute bug fix, the right way

The smallest useful loop. One main agent, one reviewer, one commit. Watch how the AGENTS.md from step 1 quietly changes the output — without any cleverness from you.

The scenario

A user reports: "When I refresh the dashboard after my session expires, I get a 500, not a 401." Reproducible. The bug is somewhere in the auth middleware. You have your stack from steps 1–3: AGENTS.md, OpenCode, reviewer agent.

stack assumed

Node + TypeScript, Fastify, Vitest, pnpm. The AGENTS.md from step 1 is in place. @reviewer is configured. Nothing else.

The flow, turn by turn

youmain session · sonnet 4.5
There's a 500 instead of a 401 when a request hits the dashboard with an expired session. Repro: POST /api/auth/login -> get cookie wait 31 minutes (session TTL is 30) GET /api/dashboard -> currently 500, should be 401 Find the bug and write a failing test before the fix.

The prompt does two things on purpose: it specifies the repro and demands a failing test first. The TDD rule is in your AGENTS.md already; the second sentence just makes it the foreground.

agentsonnet 4.5 · 14s
Reading src/server/middleware/auth.ts and the tests. The middleware throws when the session lookup returns null: const session = await sessions.get(cookie) if (session.expiresAt < Date.now()) ... ← crashes here when session is null That's the 500. Will write a failing integration test first. Added __tests__/integration/auth-expired.test.ts: it("returns 401 when session is expired", async () => { const cookie = await loginAndAdvanceClock(SESSION_TTL + 1) const res = await app.inject({ method: "GET", url: "/api/dashboard", headers: { cookie } }) expect(res.statusCode).toBe(401) }) Running it to confirm it fails for the expected reason... it does: TypeError: Cannot read properties of null (reading 'expiresAt') Now fixing src/server/middleware/auth.ts:31 to check for null before accessing expiresAt.

Two things to notice. First, the agent named the right file because of house conventions in AGENTS.md — it didn't have to search; the project structure was already loaded in its window. Second, it ran the test before claiming it was failing for the right reason. That's the difference between TDD and theater, and it's in your AGENTS.md as a rule.

agentsonnet 4.5 · 6s
Patched. Diff: const session = await sessions.get(cookie) + if (!session || session.expiresAt < Date.now()) { + return reply.code(401).send({ error: "session_expired" }) + } - if (session.expiresAt < Date.now()) { - return reply.code(401).send({ error: "session_expired" }) - } Test now passes. Should I hand it to @reviewer?

The agent asks before invoking the reviewer. That's intentional — fan-out to a subagent costs tokens, and you want a beat to look at the diff yourself. In this case, the diff is tiny and looks right.

you
Yes — @reviewer.
@reviewerhaiku-4-5 · 4.1s · fresh window
## Reviewer report - ✓ typecheck — clean - ✓ lint — clean - ✓ tests — 184 passing (1 new), 0 failing ### Notes The new test `returns 401 when session is expired` runs and asserts the exact status code. No coverage regression. Diff in src/server/middleware/auth.ts:31 is the minimal change. One observation, not a blocker: there are two other places (src/server/middleware/api-key.ts:24 and src/ws/handshake.ts:12) that do the same `session.expiresAt` dereference without the null check. They probably need the same fix.

That last paragraph is what the reviewer is for. The main agent fixed the reported bug; the reviewer noticed two structurally identical bugs the user didn't report. It didn't fix them — that's not its job — but it pointed.

you
Good catch. Fix both, same pattern, add tests for each. Then commit all three with conventional commits.
agentsonnet 4.5 · 28s
Done. Three commits: fix(auth): return 401 on expired session in dashboard middleware fix(auth): handle null session in api-key middleware fix(ws): handle null session during websocket handshake @reviewer again?

Ship it. Total: roughly nine minutes of wall clock, three commits, four test cases. You touched the keyboard four times.

What did the work, exactly

ForceWhat it contributed
AGENTS.md The agent went to the right files without searching; used Conventional Commits without being reminded; wrote tests first because the rule was in scope.
Reviewer agent Caught two structural duplicates the user didn't know about. Ran the tests in a fresh window — wasn't biased by the implementer's reasoning.
Plain prompting Naming the repro precisely. Asking for failing test first. Pausing before reviewer. None of this requires fancy framework.

The same task, without the stack

For contrast — the realistic-but-bad version of the same task:

you (no AGENTS.md)
There's a bug where session expiry returns 500 not 401. Can you fix it?
agent (no AGENTS.md)sonnet 4.5 · 22s
I'll need to look around. Where's your session middleware? I see a few possibilities... is this Express or Fastify? Are you using cookies or JWTs? What test framework should I use? I'll go ahead and patch it without tests first — let me know if you want tests added.

Same model. Same bug. Without the file, the agent burns turns on questions you've answered a hundred times — and then defaults to patching without a test, because no rule says otherwise. Multiply that by every bug-fix you do this month.

The honest comparison

Without AGENTS.md + reviewerWith the stack
Wall clock~22 min (with clarifying turns + you running tests)~9 min
Bugs foundThe one reportedThe one reported + two structurally identical
Tests addedMaybe one, if you remembered to askThree, one per fix
Commit hygieneOne squashed "fix stuff" commitThree conventional commits
Tokens (approx)~9K~14K (incl. two reviewer invocations)

The token bill is higher. The total cost — wall clock, mental load, missed bugs — is materially lower. This is the trade you're buying, every time, on small tasks. The next example shows the same pattern scaling up.

the lesson

For 90% of your day-to-day work, the right answer is exactly this: one main agent, one reviewer, a good AGENTS.md. No orchestration. No worktrees. No Ralph. The reason the rest of this guide exists is the other 10%.

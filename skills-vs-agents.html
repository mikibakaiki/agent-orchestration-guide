Skills vs. agents: what's the difference

The two setups in these field notes pick opposite primitives. Superpowers is built on skills; your colleague's setup is built on agents. They solve overlapping problems in fundamentally different ways — and once you see the distinction clearly, the choice of which to use when becomes obvious.

The one-sentence version

the distinction

A skill is knowledge injected into the current context — a procedure the same agent reads and follows. An agent is a separate context window with its own tools and permissions — work delegated to a fresh mind. Skills change what the agent knows; agents change who does the work.

What each one actually is

Skill

A markdown file describing a procedure, loaded into the current agent's context when relevant. The agent reads it and follows the steps itself. No new context window, no handoff. Think of it as a just-in-time instruction manual the agent pulls off a shelf.

Mechanism: text appended to the active context. The same model, now better-informed.

Agent

A separate invocation with its own system prompt, model, tools, and permissions. The orchestrating agent hands it a task; it works in isolation and returns a result. The parent never sees the agent's internal reasoning — only its output.

Mechanism: a fresh context window. A different worker, reporting back.

The mental model that makes it click

Imagine onboarding a new engineer.

A skill is the runbook

You hand them the document titled "How we deploy." They read it and now they know the procedure. It's their hands doing the work, but better-informed. The runbook doesn't do anything on its own — it makes whoever reads it more capable.

An agent is the specialist down the hall

You say "ask the QA engineer to run the suite." They go away, do it in their own office with their own tools, and come back with "3 failures, here's the report." You never watched them work — you got the outcome. Their mess stayed in their office.

Both are useful. The runbook is how you stop re-explaining the deploy process. The specialist is how you avoid having to become a QA engineer mid-task. The mistake is using one where you need the other.

How the two setups embody this

Superpowers does use subagents too — but its primitive, the thing you author and compose, is the skill. Your colleague's setup has procedural knowledge in its prompts too — but its primitive is the agent. Each leans on one mechanism as the unit of design.

Pros and cons

Skills

Pros

  • Cheap — no extra invocation, just context tokens
  • The agent keeps full holistic view of the task
  • Composable — many skills can apply at once
  • Easy to author: it's just a markdown procedure
  • No handoff means no nuance lost in translation
  • Portable across harnesses (markdown is markdown)

Cons

  • Everything shares one window → context rot as skills pile up
  • No permission isolation — the agent has all its tools the whole time
  • Can't route to a cheaper/different model per task
  • "Mandatory" skills can fire when they don't fit
  • No parallelism — one mind, one thing at a time

Agents

Pros

  • Fresh context window each time — directly fights rot
  • Independent permission scoping (read-only reviewer, etc.)
  • Per-task model routing (Haiku to explore, Opus to design)
  • True parallelism (the A/B critics, worktrees)
  • The parent's window stays clean — only summaries return

Cons

  • Expensive — each agent is a full invocation (the 15× tax)
  • Coordination overhead; handoffs lose nuance (MAST)
  • The orchestrator accumulates context and can itself rot
  • More machinery to maintain and debug
  • Easy to over-decompose coupled work that shouldn't be split

the heuristic

Ask: "Do I need a different brain, or just a better-informed one?" Different brain — different window, different tools, different model, or parallel — that's an agent. Same brain, just needs to know the procedure — that's a skill. If the answer is "the same agent should just know this," reaching for an agent is over-engineering. If it's "I need this done in isolation and verified," a skill won't give you the isolation.

The relationship to AGENTS.md

Where does the AGENTS.md from this guide's step 1 sit? It's the always-on skill — declarative knowledge loaded at the top of every window, every session, no trigger needed. A skill (superpowers-style) is a conditionally-loaded procedure: pulled in when relevant. So there's a spectrum:

AGENTS.md — always loaded

Your unconditional house rules and stack. Every session, top of window. The cheapest, most reliable layer. Start here.

Skills — loaded on demand

Procedures too long or too situational for AGENTS.md. "How to do a database migration here." Loaded when the task matches, then they leave. Keeps AGENTS.md lean.

Agents — dispatched when isolation pays

When you need a fresh window, different tools, a different model, or parallel work. The most powerful and most expensive layer. Add only where you can name the bottleneck (chapter 03).

The progression is cost-ordered: always-on text is nearly free, on-demand skills cost a little, agents cost a lot. Reach down the list only when the cheaper layer can't do the job. That ordering is the whole thesis of this guide, restated in primitives.

a note on tooling reality

Not every harness has first-class "skills." Claude Code has skills, subagents, and CLAUDE.md. OpenCode centers on agents + AGENTS.md. Copilot has instructions files + path-scoped instructions. The concepts are universal even where the feature names differ — an "instructions file scoped to a path" is a skill by another name; a "subagent" is an agent. Map the concept onto whatever your tool calls it.

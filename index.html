Personal AI agent orchestration

A pragmatic, skeptical guide. Fifteen standalone HTML chapters: enough theory to make defensible choices, six concrete build steps for your stack, and four worked examples that go from a ten-minute bug fix to an overnight Ralph loop in a sandbox.

the real claim

Your repetition problem — re-typing framework, code style, and git workflow into every prompt — is a context-engineering problem, not an agent problem. One good AGENTS.md solves more of it than five named "team-lead" or "architect" agents ever will. Build the boring markdown layer first. Then add agents where you can name the bottleneck they remove.

Six concrete steps for a developer using GitHub Copilot, OpenCode, VS Code, and the Copilot CLI. Each step is one HTML page with copy-able templates. Do them in order. The first one alone — your AGENTS.md — fixes most of the repetition pain.

The standardized cross-tool file. Global personal version + project version + path overrides. Templates you can copy and trim.

How to read this

If you have an hour: read 01, do step 1, read example A. That is the minimum viable version — and for many readers, it will be the maximum useful version.

If you have an afternoon: do steps 1–3, read examples A and B, skim the theory.

If you have a weekend: everything, in order. Build the full stack. Then read 15 and set a calendar reminder for ninety days from now to evaluate honestly whether it earned its keep.

a warning, up front

The published Ralph-loop success stories are almost all greenfield work. The published parallel-agent wins are mostly research breadth-first. The published multi-agent benchmarks are mostly vendor-internal. None of that makes the patterns useless. It does mean you should weight the build steps in this order: boring file > one good reviewer > worktrees > everything else.
